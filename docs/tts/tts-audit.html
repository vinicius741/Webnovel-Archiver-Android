1 Executive summary & verdict

The shortest version: the foundation is solid, but one critical correctness feature is missing and a few patterns are dated.

The current TTS subsystem is one of the most carefully engineered features in the codebase. It correctly uses a process‑wide singleton engine, a real foreground service with the right mediaPlayback type, a MediaSession for lock‑screen / Bluetooth controls, a well‑designed chunking engine, and persistent resume. These are all the hard architectural decisions, and they are made correctly.

However, measured against the current Android best‑practice benchmark, there are real gaps. The most important is that audio focus is never requested — TTS does not consistently participate in system focus behavior for phone calls, other media apps, or navigation prompts. The second tier of issues concerns robustness (silent error swallowing, no watchdog), API currency (MediaSessionCompat is legacy while androidx.media3 is the modern home), and i18n inconsistency (some TTS strings are declared but bypassed by hardcoded literals).

Overall verdict

Competent, with one must‑fix gap

Architecture is sound and idiomatic; audio focus is the single missing piece that would otherwise make this a textbook TTS implementation.

2.3 Key design choices in the current code

Threading model ("R8")

A dedicated CoroutineScope(SupervisorJob() + Dispatchers.Main.immediate) plus a Mutex serialize state mutations triggered by the async UtteranceProgressListener callbacks.

Doc vs. reality mismatch

The KDoc claims public control methods (play/next/previous/pause/stop) are routed through stateMutex, but only onInit and onDone actually acquire it. Works only because callers are currently on the main thread.

State propagation

A hand‑rolled multicast observer registry (stateListeners) — no StateFlow/LiveData/ViewModel. Each state change fans out a TtsPlaybackSnapshot (or null).

Chunking & highlight invariant

prepareTtsChunks() and prepareTtsAnnotatedHtml() walk the same elements with the same grouping rule, so the spoken chunks are byte‑for‑byte aligned with the highlighted DOM groups. This is the foundation of the highlight feature.

Persistence & resume

Atomic JSON files (tts_settings.json, tts_session.json) wrapped with appVersion envelopes. Resume is checked at app launch and from the notification, with chunk‑index clamping and a restored‑chunk‑size floor.

3 The modern Android benchmark

What "best practice" looks like for a long‑form narration / TTS app on Android 15/16, drawn from current official Android documentation.

3.1 Audio focus — mandatory for any app that produces sound

Per the official Manage audio focus guide, "only one app can hold audio focus at a time. When your app needs to output audio, it should request audio focus, and only play sound when it has it." For long‑form speech playback the correct request is AUDIOFOCUS_GAIN (or TRANSIENT_MAY_DUCK for short bursts), with an OnAudioFocusChangeListener that handles:

  • LOSS_TRANSIENT → pause TTS, remember position
  • LOSS_TRANSIENT_CAN_DUCK → for spoken content, usually pause rather than duck, using setWillPauseWhenDucked(true)
  • LOSS → stop playback entirely
  • GAIN → resume from saved position

The AudioAttributes should declare USAGE_MEDIA with CONTENT_TYPE_SPEECH for long‑form narration. Since this project targets SDK 36, the Android 15+ focus rule also matters: focus requests can fail unless the app is the top app or running a foreground service, so request focus from the foreground playback path and handle AUDIOFOCUS_REQUEST_FAILED.

3.2 Foreground service — correct type & timing

  • Declare a foreground service type — mediaPlayback is the right one for TTS (required since Android 14 / API 34).
  • Hold both FOREGROUND_SERVICE and the subtype permission FOREGROUND_SERVICE_MEDIA_PLAYBACK.
  • Call startForeground() promptly after startForegroundService() (within the ANR window).
  • Keep the notification ongoing only while there is an active session.

3.3 MediaSession — prefer androidx.media3

The legacy MediaSessionCompat (from androidx.media:media, which this project uses at 1.7.0) is now documented under Android's legacy media APIs. The modern, actively developed home is androidx.media3 (1.10.1 stable as of the 2026‑06‑24 AndroidX release notes, with 1.11.0 alpha in preview), which offers media3-session, MediaSessionService, MediaController, and official View/Compose media UI artifacts. For new media apps Media3 is the recommendation; for an existing working service, MediaSessionCompat remains functional but is no longer where new Jetpack media investment is concentrated.

3.4 UtteranceProgressListener — error handling & reliability

  • Set the listener via setOnUtteranceProgressListener before calling speak().
  • Pass a unique utteranceId per chunk so callbacks fire correctly.
  • Override the API 21 onError(utteranceId, errorCode) variant (the old single‑arg form is deprecated) and react to ERROR_* codes.
  • Separately check setLanguage/setVoice results and language availability; LANG_MISSING_DATA and LANG_NOT_SUPPORTED are language/voice setup outcomes, not onError error codes.
  • Guard against destroyed views in callbacks (they can arrive after onDestroy).
  • Consider a watchdog: if onDone/onError never arrives for a chunk, the engine can stall silently.

3.5 Memory & streaming

For long chapters, stream/chunk text into the TTS queue rather than synthesizing the whole document at once — which the current code already does well.

5 What the implementation does well

Worth preserving when any refactor happens.

Architecture
  • Clean separation of pure logic (*Planning.kt, TtsPlaybackState) — all unit‑tested — from Android‑bound code.
  • A single process‑wide TtsEngine via a manual service locator avoids the classic "activity and service each spin up a TextToSpeech" race.
  • The chunk‑list ↔ annotated‑HTML 1:1 alignment invariant is elegant and explicitly documented.
Correctness & resilience
  • Foreground promotion with the right mediaPlayback type and a flag to prevent redundant startForeground calls.
  • STOP_FOREGROUND_REMOVE + flag reset so a later ACTION_START in the same process re‑enters foreground correctly.
  • Ongoing notification while a session exists (playing or paused); dismissable only on stop.
  • Idempotent, identity‑based addStateListener; defensive copy + runCatching around listener dispatch.
  • Double‑layer validation: clamping at input (SettingsValidation) and normalization at persistence (PreferenceNormalization).

6 Gaps & risks (detailed)

The specific deviations from best practice, with evidence and impact.

Gap 1 — No audio focus management Critical

Evidence: rg -n "AudioFocus|requestAudioFocus|AudioManager|AUDIOFOCUS|AudioAttributes" android/app/src/main returns zero matches.

Impact: TTS does not reliably pause, resume, or get system-muted for incoming calls, other media apps (Spotify, YouTube), or navigation prompts because Android's enforced focus behaviors depend on a successful focus request. On Android 15+ / target SDK 35+, focus requests also fail unless made while the app is topmost or running a foreground service. For a mediaPlayback foreground service this is non‑idiomatic and a real correctness/UX issue.

Fix shape: Introduce an AudioFocusRequest with AUDIOFOCUS_GAIN, setWillPauseWhenDucked(true), and AudioAttributes (USAGE_MEDIA + CONTENT_TYPE_SPEECH), requested from the foreground playback path before speakCurrent() and abandoned on stop/finishPlayback. Wire an AudioManager.OnAudioFocusChangeListener into pause/resume/stop, and do not call speak() if focus is denied.

Gap 2 — Silent error swallowing High

Evidence: TtsEngine.kt:84 — override fun onError(utteranceId: String?) = Unit. The deprecated single‑arg form is overridden; the API‑21 onError(String, Int) with the error code is not. onStart is also a no‑op.

Impact: Engine‑busy, language‑missing‑data, and synthesis failures are invisible. Playback simply stops advancing with no user feedback or telemetry. A user whose selected voice's language data isn't installed gets a dead feature with no explanation.

Correction: LANG_MISSING_DATA/LANG_NOT_SUPPORTED should be handled from language/voice setup paths such as setLanguage and setVoice, while onError(String, Int) reports TextToSpeech.ERROR_* synthesis failures.

Gap 3 — Mutex documentation overstates the guarantee Medium

Evidence: The class KDoc (TtsEngine.kt:30‑35) claims public control methods (play/pause/next/...) are routed through stateMutex, but none of them acquire it — only onInit (L76) and onDone (L89) do.

Impact: Today it works because all callers are on the main thread and Dispatchers.Main.immediate keeps the onDone continuation on the same thread. It is a latent race the moment any caller invokes a control off‑main, and the comment misleads future maintainers.

Gap 4 — No stall watchdog Medium

Evidence: Advancement depends entirely on onDone firing. If the engine hiccups and neither onDone nor onError arrives for a chunk, playback stalls with no recovery and no signal to the user.

Gap 5 — i18n strings declared but bypassed High

Evidence: strings.xml defines tts_channel_name, tts_notif_title, tts_notif_paused, etc. (L21–27), but TtsForegroundService.buildNotification (L226–230), updateMediaSessionMetadata (L382–384), and createNotificationChannel (L396) hardcode English literals ("Text to Speech", "TTS paused", "Reading", …). TtsNotificationActions.kt also hardcodes "Previous", "Play", "Pause", "Next", and "Stop".

Impact: The i18n plumbing exists but is incomplete for TTS; non‑English users see English notification text, action labels, and system media metadata fallbacks.

Gap 6 — Legacy media stack Medium

Evidence: The app depends on androidx.media:media:1.7.0 (MediaSessionCompat). Android's current Media3 release notes list androidx.media3 1.10.1 stable as of 2026‑06‑24, and the official guides put compat APIs under "Legacy media APIs" while Media3 provides media3-session, MediaSessionService, and Compose media UI artifacts.

Impact: No functional regression today, but the project is investing in a stack that is no longer receiving new features, and Compose‑based media UI (when the reader eventually moves to Compose) will be smoother on Media3.

Gap 7 — Minor: eager init, dead fields, hidden voice metadata

  • Eager engine construction (TtsEngine.kt:38) binds the system TTS service at process start even if the user never uses TTS. Lazy init on first play() would save resources.
  • sessionVersion on TtsSession is forward‑looking but currently unread — no migration logic consumes it.
  • VoiceInfo.quality/latency are surfaced in the DTO but the picker UI doesn't display them — a missed UX cue (let users avoid low‑quality/network voices).

7 Recommendations (prioritized)

Ordered by impact. Each item is a candidate for a future task — no application code is changed in this investigation.

  1. Critical Add audio focus management.

    Introduce AudioFocusRequest (AUDIOFOCUS_GAIN) with speech‑oriented AudioAttributes, requested at playback start and abandoned at stop. Implement an OnAudioFocusChangeListener that pauses on transient loss, pauses rather than ducks for spoken-content duck requests, fully stops on permanent loss, and resumes on regain when appropriate. This is the single change that closes the biggest correctness gap and aligns with the official Manage audio focus guidance.

  2. High Implement real utterance error handling.

    Override the API‑21 onError(utteranceId: String?, errorCode: Int) form and surface TextToSpeech.ERROR_* failures to the user and/or logs. Separately check setLanguage/setVoice results and voice availability, and for LANG_MISSING_DATA offer to launch the voice‑data installer intent. Without this, the feature fails silently for a class of users.

  3. High Use the declared i18n strings in the notification.

    Replace the hardcoded literals in buildNotification, updateMediaSessionMetadata, createNotificationChannel, and TtsNotificationActions with getString(R.string.tts_...) or resource IDs resolved by the service. Some strings already exist; add the missing action-label and metadata-fallback strings in the same pass.

  4. Medium Reconcile the stateMutex contract with the code.

    Either (a) actually wrap the public control methods in scope.launch { stateMutex.withLock { ... } } so the documented guarantee holds, or (b) document that callers must be on the main thread and drop the mutex claim. Option (a) is safer and removes a latent race.

  5. Medium Add a playback‑stall watchdog.

    Schedule a coroutine that, if no onDone/onError arrives for the current chunk within a generous timeout (e.g. 2× expected duration), re‑issues speakCurrent() or surfaces an error. Prevents the "TTS silently stops mid‑chapter" failure mode.

  6. Medium Evaluate migrating to androidx.media3.

    Not urgent — the current MediaSessionCompat setup works — but worth planning, especially if/when the reader moves to Compose (Media3's current stable line includes official Compose media‑UI artifacts). Treat as a deliberate, scoped migration, not a drive‑by change.

  7. Low Lazy‑initialize the TextToSpeech engine.

    Move construction out of the field initializer into a ensureEngine() called on first play()/playFromChunk()/resumePersistedSession(), so the system TTS service only binds when the user actually uses TTS.

  8. Low Surface voice quality/latency in the picker.

    The DTO already carries quality/latency; show them so users can avoid low‑quality or network‑dependent voices. Also refresh the voice list when the engine finishes initializing (currently the picker only toasts "No local TTS voices available yet" if opened too early).

  9. Low Add coverage for chunk navigation & error paths.

    Current unit tests cover only the pure *Planning/TtsPlaybackState helpers. Add tests for next()/previous() index arithmetic (subtle due to the post‑increment + clamps) and for the new error‑handling branches once implemented.

8 Explicitly out of scope

As requested, no application source code was modified for this investigation. Everything above is findings and recommendations only. Any code changes would be separate, explicitly‑requested tasks.

The following were intentionally not pursued and would need their own investigation if desired:

  • A cloud/remote TTS engine (e.g. a neural voice SDK) — the app is deliberately built on the on‑device Android TextToSpeech engine.
  • Per‑character / word‑level highlighting (the current highlight is per chunk/group).
  • SSML support (Android's TextToSpeech engine does not natively support SSML).
  • Android Auto / Automotive surface (the MediaSession is in place, but Auto‑specific metadata/branding was not reviewed).
