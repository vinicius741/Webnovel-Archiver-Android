Master Executive Summary

ScribbleHub requests fail in OkHttp because Cloudflare blocks the request at multiple layers: TLS/JA4 fingerprinting, missing JS challenge execution, a stale User-Agent (Chrome/114), and no cookie persistence. The existing "Open in Browser" fallback uses a Chrome Custom Tab, which operates in Chrome's process and isolates its cookies—meaning the app's OkHttp client never receives the earned cf_clearance cookie.

The Definitive Solution (The Mihon Pattern): Bridge OkHttp and Android's native android.webkit.CookieManager via a custom AndroidCookieJar. When a Cloudflare challenge is encountered, solve it using an in-app WebView (via a background transient solver interceptor or an interactive SourceAccessActivity), capture the cf_clearance cookie directly from CookieManager, and replay it on OkHttp requests with an identical User-Agent.

1. Detailed Root Cause Analysis

ScribbleHub novel metadata fetching, chapter downloads, and table-of-contents AJAX calls are currently failing when Cloudflare presents a challenge. A thorough audit of the codebase reveals two core architectural flaws causing this failure:

A. Why the OkHttp Client is Rejected

  • TLS Fingerprint (JA3/JA4): OkHttp uses Android's Conscrypt/BoringSSL TLS stack. Cloudflare evaluates the cipher suite order and extension fingerprint in the initial ClientHello packet before any HTTP headers are read. OkHttp's fingerprint immediately identifies it as automated non-browser traffic.
  • HTTP/2 Settings & Frame Ordering: HTTP/2 frame sizes, window updates, and stream priorities differ between standard OkHttp and desktop/mobile Chrome.
  • Lack of JS Execution: Cloudflare challenges deliver a JavaScript payload from /cdn-cgi/challenge-platform/.... OkHttp cannot execute JS, so it returns the raw challenge HTML page instead of the content.
  • No Cookie Persistence: NetworkClient.defaultClient in source/network/NetworkClient.kt is constructed without a CookieJar. Even if a response returns a Set-Cookie header, OkHttp drops it immediately.
  • Stale User-Agent String: NetworkRequests.kt hardcodes Chrome/114.0.0.0. Modern Cloudflare security rules score outdated browser UAs as high risk.

B. The Chrome Custom Tab Trap

Why the current fallback fails

In feature/browser/SourceAccessActions.kt, when a challenge is detected, the app calls showBrowser() to launch a Chrome Custom Tab. Custom Tabs run inside Chrome's separate process and write cookies exclusively to Chrome's private app data sandbox. There is no Android API for third-party apps to extract cookies from Chrome Custom Tabs. When the user returns to the app, OkHttp retries the request with an empty cookie jar and is blocked again immediately.

2. Cloudflare 2026 Threat Model

Cloudflare evaluates incoming requests through a multi-tiered security pipeline. Understanding these layers dictates why on-device WebView cookie capture is required:

Layer Mechanism OkHttp Behavior WebView Behavior
1. IP & ASN Reputation Evaluates IP subnet (datacenter vs mobile/residential). Depends on user's connection. Uses user's native mobile/Wi-Fi IP (High Trust).
2. TLS (JA4) Fingerprint Hashes ClientHello ciphers & extensions. Identified as Java/OkHttp (Blocked). Matches Chromium native TLS (Passes).
3. HTTP/2 Fingerprint Inspects HTTP/2 frames and window parameters. Non-browser pattern (Blocked). Native Chromium H2 (Passes).
4. JS Challenge / Turnstile Executes browser environment checks (canvas, WebGL, timing). Cannot run JS (Blocked). Executes full Chromium JS environment (Passes).
5. Session Binding Binds cf_clearance to (IP + User-Agent + JA4). No cookie jar to store (Lost). Stored in CookieManager (Persisted).

4. Mihon Reference Architecture

The maintained manga reader Mihon (formerly Tachiyomi) represents the gold standard for Cloudflare bypass on Android. Its architecture relies on four pillars:

  1. Unified Cookie Bridge (AndroidCookieJar): Implements OkHttp's CookieJar interface by delegating directly to android.webkit.CookieManager.getInstance(). Any cookie set in a WebView is immediately available to OkHttp calls without explicit copy operations.
  2. Transient Background Interceptor (CloudflareInterceptor): When an HTTP response returns status 403/503 with Cloudflare challenge markers, the interceptor clears any existing stale cf_clearance cookie, instantiates an off-screen WebView on the main thread, loads the URL, and waits up to 30 seconds for a fresh cf_clearance cookie to appear in CookieManager.
  3. Strict User-Agent Synchronization: The interceptor sets the WebView's User-Agent to match the exact UA header sent by OkHttp. Cloudflare rejects clearance cookies if the request UA does not match the solving UA.
  4. Interactive UI Fallback: If the background WebView cannot solve the challenge (e.g. an interactive Turnstile CAPTCHA is presented), the app prompts the user to open an in-app WebViewActivity to solve it manually.

6. Kotlin Technical Specifications

Below are the production-grade Kotlin implementation specifications tailored for this project:

A. AndroidCookieJar.kt

package com.vinicius741.webnovelarchiver.source.network

import android.webkit.CookieManager
import okhttp3.Cookie
import okhttp3.CookieJar
import okhttp3.HttpUrl

class AndroidCookieJar : CookieJar {
    private val cookieManager: CookieManager by lazy { CookieManager.getInstance() }

    override fun saveFromResponse(url: HttpUrl, cookies: List<Cookie>) {
        val urlString = url.toString()
        for (cookie in cookies) {
            cookieManager.setCookie(urlString, cookie.toString())
        }
        cookieManager.flush()
    }

    override fun loadForRequest(url: HttpUrl): List<Cookie> {
        val cookieString = cookieManager.getCookie(url.toString()) ?: return emptyList()
        if (cookieString.isBlank()) return emptyList()

        return cookieString.split(";").mapNotNull { pair ->
            Cookie.parse(url, pair.trim())
        }
    }
}

B. CloudflareBypassInterceptor.kt

package com.vinicius741.webnovelarchiver.source.network

import android.content.Context
import android.os.Handler
import android.os.Looper
import android.webkit.CookieManager
import android.webkit.WebView
import android.webkit.WebViewClient
import okhttp3.Interceptor
import okhttp3.Request
import okhttp3.Response
import java.util.concurrent.CountDownLatch
import java.util.concurrent.TimeUnit

class CloudflareBypassInterceptor(
    private val context: Context,
    private val defaultUserAgent: String
) : Interceptor {

    private val mainHandler = Handler(Looper.getMainLooper())

    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request()
        val response = chain.proceed(request)

        // Check if response is a Cloudflare challenge
        if (response.code in listOf(403, 503) && isChallengeResponse(response)) {
            response.close() // Close initial failed response body

            val solved = resolveWithWebView(request)
            if (solved) {
                // Retry request with newly acquired cf_clearance cookie
                return chain.proceed(request)
            }
        }
        return response
    }

    private fun isChallengeResponse(response: Response): Boolean {
        val serverHeader = response.header("Server") ?: ""
        val cfMitigated = response.header("cf-mitigated") ?: ""
        if (cfMitigated.equals("challenge", ignoreCase = true)) return true
        if (serverHeader.contains("cloudflare", ignoreCase = true)) {
            val peekBody = response.peekBody(1024 * 64).string()
            return peekBody.contains("cf-challenge") || peekBody.contains("just a moment", ignoreCase = true)
        }
        return false
    }

    private fun resolveWithWebView(request: Request): Boolean {
        val latch = CountDownLatch(1)
        var isSuccess = false
        val url = request.url.toString()
        val cookieManager = CookieManager.getInstance()

        mainHandler.post {
            val webView = WebView(context).apply {
                settings.javaScriptEnabled = true
                settings.domStorageEnabled = true
                settings.userAgentString = request.header("User-Agent") ?: defaultUserAgent
                
                webViewClient = object : WebViewClient() {
                    override fun onPageFinished(view: WebView?, url: String?) {
                        val cookies = cookieManager.getCookie(url) ?: ""
                        if (cookies.contains("cf_clearance")) {
                            isSuccess = true
                            latch.countDown()
                        }
                    }
                }
            }
            webView.loadUrl(url)

            // Destroy WebView after timeout
            mainHandler.postDelayed({
                webView.stopLoading()
                webView.destroy()
                latch.countDown()
            }, 30000)
        }

        latch.await(30, TimeUnit.SECONDS)
        return isSuccess
    }
}

7. Step-by-Step Implementation Roadmap

Phase 1: Foundation — Shared Cookie Jar

Step 1

Create AndroidCookieJar.kt in source/network/. In AppContainer.kt, instantiate AndroidCookieJar and wire it to NetworkClient.defaultClient via .cookieJar(...). Enable CookieManager.getInstance().setAcceptCookie(true) on app startup.

Phase 2: User-Agent Synchronization & Interceptor Wiring

Step 2

Update NetworkRequests.kt to infer modern Chrome UA from WebSettings.getDefaultUserAgent(context). Add CloudflareBypassInterceptor to NetworkClient.defaultClient. Ensure OkHttp UA and WebView UA are 100% identical.

Phase 3: Interactive In-App WebView Screen

Step 3

Create CloudflareSolveActivity.kt under feature/browser/. Replace Chrome Custom Tab invocation in SourceAccessActions.kt with CloudflareSolveActivity. When opened, it loads ScribbleHub, detects when cf_clearance is present in CookieManager, flushes cookies, and finishes with a successful result code.

Phase 4: Defensive Throttling & RSS Payload Optimization

Step 4

Add source rate-limiting in ScribbleHubProvider.kt (minimum 1.5s–3s gap between requests). For novel update checks, switch to polling rssfeed.php under the active cf_clearance cookie to reduce HTML crawl overhead.

Phase 5: Diagnostics & Cookie Management Settings

Step 5

Add "Clear ScribbleHub Cookies" and "WebView System Info" options in Settings so users can clear stale clearance tokens or verify their installed WebView package version if issues arise.

9. Compliance & Security Boundary

Legal & Ethical Posture

This strategy strictly adheres to user-mediated first-party session management. It does not reverse-engineer security algorithms, use third-party CAPTCHA solving services, or deploy proxy rotation networks. The user authenticates and solves challenges on their own device inside a standard Android Chromium WebView context to read their personal novel library.
