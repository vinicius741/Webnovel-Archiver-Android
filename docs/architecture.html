How Webnovel Archiver is Built

A single-activity, no-XML, no-DI-framework Android app that archives web novels — fetching story metadata & chapters from Royal Road and ScribbleHub, cleaning the text, packaging EPUBs, reading aloud with TTS, and surviving crashes & restores. This document is an interactive walkthrough of every layer, engine, and data flow.

00At a Glance

The app is organized as a classic layered architecture, but with a few deliberate departures from "stock" Android. The code is split into pure, unit-testable planning objects and stateful engines/services — and the UI is built entirely with a programmatic View DSL.

🎯 Guiding Principles

  • Planning vs Engine split. Deterministic logic lives in pure *Planning objects (no Android imports); I/O & coroutines live in *Engine classes.
  • Single owner. One AppStorage, one NetworkClient, one of each engine — held by a manual service locator, not a DI framework.
  • No XML UI. Every screen is built with a programmatic DSL of extension functions on ViewGroup.
  • No WorkManager. Background work is foreground services with self-contained coroutine loops (START_STICKY).
  • Durability-first storage. Atomic writes + fsync; relative paths; transactional restores.

03App & Lifecycle Layer

The thinnest layer: it wires process-wide dependencies, owns the single Activity, and defines the ScreenHost contract that every screen implements against.

service locator AppContainer — the manual DI

Instead of a DI framework, AppContainer is a plain class holding exactly one instance of each process-wide dependency. It's created in WebnovelArchiverApp.onCreate() and accessed via the Context.appContainer extension. The whole point: prevent the Activity and the two foreground services from instantiating duplicate engines that race on the same JSON files.

class AppContainer(context: Context) {
    val applicationScope = CoroutineScope(SupervisorJob() + Dispatchers.IO)
    val network: NetworkClient = NetworkClient()
    val storage: AppStorage = AppStorage(context)
    val repository = AppRepository(storage)
    val syncEngine = StorySyncEngine(storage, network)
    val epubEngine = EpubEngine(storage, network)
    val ttsEngine = TtsEngine(context.applicationContext, storage)

    fun init() {                          // called once from Application.onCreate
        storage.migrateChapterPathsToRelative()
        storage.recoverInterruptedDownloads()
        repository.refresh()
    }
}

MainActivity — lifecycle & wiring only

The Activity declares all configChanges in the manifest (orientation, screenSize, uiMode, smallestScreenSize…) so Android does not recreate it on rotate/fold/theme change. It implements ScreenHost, pulls dependencies from the container, sets up the root FrameLayout, and hands the first screen off. All rendering & navigation live in extension functions elsewhere.

onCreate responsibilities

  1. Pull storage/engines from appContainer
  2. Build the Activity's own DownloadEngine (enqueue/control only)
  3. Apply theme & edge-to-edge window
  4. Create frame root + register back callback
  5. Start FoldTracker & re-render on fold change
  6. Resolve resume target (TTS session) or browser import, else showLibrary()

Lifecycle hooks

  • onConfigurationChanged → calls rerender?.invoke() to reflow the live screen
  • onNewIntent → handles browser import URLs
  • onDestroy → disposes WebViews & detaches the reader TTS listener
▶ ScreenHost — the contract every screen uses interface

ScreenHost is an interface that exposes shared dependencies (storage, repository, all engines, foldTracker), the root frame, and transient UI state (activeStory, storyOperation, etc.). Screens are internal fun ScreenHost.showXxx() extension functions — Kotlin's implicit-receiver chain lets them call showDetails(id) unqualified from inside nested layout blocks.

Key members: backHandler (set by screen(), drives hardware back), rerender (rebuilds current screen on fold/rotate), screenObserver (a Job cancelled on every navigation).

▶ Theming, foldables & live updates UI systems

Theming (4 hand-built themes)

ThemeManager is a process singleton — the single source of truth read by every view builder. Four themes ported from the legacy RN app: obsidian (dark/gold), midnight (dark/blue), forest (dark/green), classicLight (cream). Dark mode is a property of the active theme, not system-driven (the reader has a separate per-document dark override).

Foldable / responsive layout

FoldTracker uses androidx.window to detect a folding feature. ScreenLayoutPlanning.resolveScreenLayout() is a pure function mapping width/height/fold/mode → width class (COMPACT/MEDIUM/EXPANDED) and column count. The Activity declares all configChanges, so it observes the fold sensor and calls rerender to reflow the live screen.

Live updates without rebuilding

Library & Details collect repository.downloadStateVersion (a monotonic counter) in a screenObserver job, diff state, and patch views in place rather than rebuilding the whole tree. screen() cancels the observer on every navigation.

▶ Reader WebView & HTML sanitization security

The reader renders chapter content in a WebView. Before display, the HTML is passed through ChapterHtmlSanitizer.sanitize() — a Jsoup Safelist.relaxed() baseline that strips <script>, on* handlers, risky inline styles, and iframe/object/embed, while restricting a href to http/https/mailto with rel="nofollow noopener".

JavaScript is enabled only because the HTML was sanitized. The only script that runs is the one the app injects — a tiny WnaTts.setActive(i) bridge for TTS paragraph highlighting. File/content access stays locked. The bridge (ReaderTtsBridge) exposes a single onTtsStart(index) method for double-tap-to-read-from-paragraph.

05Source & Network — How Novels Are Fetched

Three types form the fetching seam: NetworkClient (the single HTTP chokepoint), SourceProvider (the per-site strategy), and SourceRegistry (the dispatch table).

SourceProvider — the per-site contract

A small, mostly-synchronous interface. The only suspend method is getChapterList, because ScribbleHub pages its table of contents via WordPress AJAX.

interface SourceProvider {
    val name: String;  val baseUrl: String
    fun isSource(url: String): Boolean
    fun getStoryId(url: String): String          // stable namespaced id
    fun getChapterId(url: String): String?
    fun parseMetadata(html): NovelMetadata      // pure, no network
    suspend fun getChapterList(html, url, network, progress): List<ChapterInfo>
    fun parseChapterContent(html): String    // throws on failure → retryable
}

Registered sources

RoyalRoad RoyalRoadProvider

  • ID: rr_<fictionId>
  • Static TOC: .chapter-row anchors (no pagination)
  • Content: .chapter-inner; strips portlets/scripts
  • Ungated — no per-host rate limit

ScribbleHub ScribbleHubProvider

  • ID: sh_<seriesId> (fallback sh_url_…)
  • Paginated TOC: POSTs to wp-admin/admin-ajax.php with X-Requested-With: XMLHttpRequest
  • Content: #chp_raw; strips author notes/sharedaddy
  • Gated: 1500ms gap between requests
▶ NetworkClient — HTTP, rate limiting & retry shared client

A thin suspend wrapper over one shared OkHttpClient (connect 30s, read 45s, no cache), constructed once in AppContainer. Three methods: fetch (text), postForm (ScribbleHub AJAX), fetchBytes (cover images, returns null on non-image/oversize, never throws).

Per-host rate limiting

A ConcurrentHashMap<host, HostGate> where each HostGate holds a Mutex + nextAllowedAt. Only www.scribblehub.com is gated (1500ms gap); all other hosts run ungated. The mutex is held only while computing the gap, so concurrent Activity + service fetches can't bypass it.

Retry policy

Up to 3 attempts, but only for ScribbleHub on HTTP 403/429, with backoff 2.5s × attempt. Any other non-2xx throws immediately. A hardcoded Android Chrome mobile User-Agent is sent on every request; no cookies, no Referer.

▶ HTML parsing & the Patreon stats subsystem Jsoup + Gson

Jsoup everywhere, regex only for IDs

Metadata parsing follows a consistent "try specific selector → fall back to OG/Twitter meta tags" pattern. Notable helper: blockText() walks the DOM inserting \n\n around block tags so the synopsis stays readable (Jsoup's .text() collapses everything to one line). Chapter content is returned as cleaned inner HTML, fed later to both the reader WebView and the EPUB.

Patreon discovery & estimation

During parseMetadata, findPatreonUrl() scans a[href*=patreon.com] links, filtering out any inside comment subtrees, and scores them by context (author link = 3, "support" anchor = 2, description = 1). The winner is stored on NovelMetadata.patreonUrl.

PatreonStatsFetcher then scrapes the creator's /about page: it walks Patreon's embedded JSON (Gson JsonParser) extracting paid_member_count and, when show_earnings, the exact pledge sum; otherwise it estimates from tier prices (median × members) with a conservative 0.9 net-of-fees multiplier. Non-USD amounts are converted via the public Frankfurter API. A regex fallback covers fields the JSON walk misses. All wrapped in runCatching so a Patreon failure never breaks a sync.

Key seam: the sync engine fetches metadata + TOC + Patreon; the download engine fetches individual chapter bodies. Both reach the source exclusively through these three types — never via raw HTTP.

06Domain & Models

Pure Kotlin — zero Android imports. Holds the persisted data models and the deterministic business rules.

📦 Core model: Story (the central aggregate)

data class Story(
    // identity
    val id, title, author, coverUrl?, description?, sourceUrl,
    // download progress
    status: DownloadStatus, totalChapters, downloadedChapters,
    chapters: MutableList<Chapter>, lastUpdated?, dateAdded?,
    // EPUB bookkeeping
    epubPath?, epubPaths?, epubStale?, epubConfig?,
    // workflow
    pendingNewChapterIds?, tags?, lastReadChapterId?, score?, tabId?,
    // archive snapshots
    isArchived?, archiveOfStoryId?, archivedAt?, archiveReason?,
    // Patreon
    patreonUrl?, patreonStats?
)

Fields are var (mutable) by design — they round-trip through Gson to JSON on disk. A chapter is id, title, url, content? (in-memory), filePath? (persisted), downloaded. Chapter HTML is stored as separate files, not embedded in the story JSON.

🗄 Archive snapshots

An archive is itself a Story row with isArchived=true and archiveOfStoryId=<original>. ArchiveSnapshotPlanning.buildArchiveSnapshot() synthesizes a frozen, file-copied record of a story at a moment in time, with a random suffix in its id. Created before the live story is overwritten on sync, so removed chapters stay readable.

🛡 Action guards & bookmarks

StoryActionGuards gates actions on archives (canSync, canQueueDownloads → false if archived). StoryBookmarkPlanning.withBookmark() sets lastReadChapterId and re-derives the EPUB range ("start after bookmark" begins on the chapter after the bookmark).

07Data & Persistence

No SQLite, no Room, no SharedPreferences. Everything is file-based JSON under filesDir/webnovel_archiver/, with two durability primitives making every write crash-safe.

📁 On-disk layout

webnovel_archiver/ # root ├── stories/ # one JSON per story │ └── rr_123.json ├── novels/ <storyId>/ # chapter HTML files │ └── 0001_Chapter_Title.html ├── epubs/ <storyId>/ # generated EPUBs (regenerated on demand) ├── backups/ # exported backups ├── library_index.json # ordered list of story IDs ├── settings.json download_queue.json ├── display_preferences.json tabs.json ├── tts_session.json tts_settings.json └── regex_cleanup_rules.json sentence_removal.json

Chapter paths are stored relative to root — so backups, restores, and app-data moves stay portable. Legacy absolute paths are migrated on read.

🔐 Durability primitive #1 — DurableJson

Used for every JSON document. Wraps Android's AtomicFile: writes to a temp file, then renames. Uniform envelope {schemaVersion, appVersion, payload} with backward-compatible reads of legacy bare-payload files. Serializer is Gson with pretty-printing.

🔐 Durability primitive #2 — AtomicFileWrites

For binary/streamed files (chapter HTML, EPUBs). Classic write-temp → fsync → atomic rename. A sibling temp file with a monotonic counter avoids collisions; RandomAccessFile.fd.sync() forces data to disk before the rename, so a post-rename crash can't lose the file.

▶ AppRepository — the caching layer with StateFlow R2 single owner

AppRepository is the single owner of AppStorage. It wraps raw file I/O with cached, observable state so UI screens don't re-parse JSON on every render (disk reads off the render path).

Holds StateFlows: libraryFlow, queueFlow, settingsFlow, and a monotonic downloadStateVersion counter (because queue/story models are mutated in place, so equality is unreliable — a counter is a safer change signal). The download engine pushes updates back via publishDownloadState(), which re-reads from storage under the shared storage monitor and bumps the version.

Transactional library/queue mutations are suspend and synchronized(storage) — the same monitor AppStorage uses, so UI, foreground service, and download engine never interleave.

▶ Backup & restore — two formats, transactional restore backup/

JSON backup (metadata only)

Exports library + tabs as JSON with chapter content scrubbed. Import merges into the existing library by ID, preserving local downloaded state for chapters already present. Capped at 50 MiB.

Full backup (ZIP with chapter files + all config)

A manifest.json plus one entry per downloaded chapter file. Paths are percent-encoded so they're safe inside a ZIP across platforms.

Transactional full restore

A staged swap: extract with limits (200k entries, 2GB total, zip-slip protection) → validate manifest → scrub transient state → build a staged replacement tree → verify it → atomic rename the old root to a cache snapshot and the staged tree into place. Because the existing root is snapshotted before the single-rename swap, a failed or interrupted restore leaves the previous library untouched.

Startup recovery: recoverInterruptedDownloads() flips any job stuck in downloading (process was killed mid-download) back to pending on the next cold start, reading/writing the queue file directly inside the synchronized block so a concurrent writer can't interleave.

DownloadEngine Fetches chapter bodies & writes them to disk

A foreground service + self-contained coroutine loop (no WorkManager). The Activity and the service each instantiate their own DownloadEngine, but they share one AppStorage — the monitor serializes them.

Per-chapter hot path

  1. Resolve provider + network.fetch(url)
  2. Parse + clean in one shot: CleanupEngine.shared.applyDownload(parseChapterContent(html), …)
  3. storage.saveChapter(...) (atomic write)
  4. Under synchronized(storage): set filePath, downloaded=true, recompute counts, mark epubStale=true

The process loop

  1. Fail unsupported-source jobs each iteration
  2. Under monitor: read queue, selectEligibleJobs (global + per-source concurrency caps), flip to Downloading
  3. Fan out: launch each job, join all before next batch
  4. Update per-source rate-limit bookkeeping

The foreground-service handshake: to satisfy Android O+'s 5-second startForegroundService→startForeground window while the durable queue write may be slow: the UI sends ACTION_PREPARE (immediate foreground with placeholder) → enqueues off the main thread → sends ACTION_START; on failure, abortPrepare rolls back.

Retry: exponential backoff (3s × 2^attempt, cap 60s), auto-retry up to maxRetries for classified-retryable errors (403/429, 5xx). Cross-process resume via recoverInterruptedDownloads() on cold start.

TtsEngine Reads chapters aloud with MediaSession + lock-screen controls

One process-wide engine shared by the Activity (reader transport + highlight) and the foreground service (MediaSession + notification). Sharing one instance means the reader's listener fires for playback the service drives.

Concurrency (R8)

Constructed on Dispatchers.Main.immediate; all state mutations (UtteranceProgressListener callbacks and control methods) serialize through a Mutex — so Android's TTS callbacks from any thread can't race storage writes or playback continuation.

Multicast observation

A list of state listeners; both the service and reader register. notifyStateListeners() fans out snapshots — push-based, never polling.

Resume across restarts: TtsSession is persisted to JSON on every chunk boundary. On launch, resumePersistedSession() rebuilds chunks and continues from the bounded index. Auto-advance: when a chapter finishes, handleChapterFinished() rebuilds the next chapter's chunks and continues seamlessly.

The foreground service owns MediaSessionCompat (transport controls, headset/Bluetooth media buttons via MediaButtonReceiver) and a MediaStyle notification with Prev/Play-Pause/Next/Stop.

09Sync Engine — in detail

See the Engines ▸ Sync tab for the visual flow. Key details below.

mergeChapters — the reconciliation core

Keyed by stable id = provider.getChapterId(url). For each incoming chapter: if it matches an existing one, it survives (preserving downloaded/filePath) but refreshes title/url; otherwise it's new (downloaded=false). Survivors are removed from a "remaining" set; whatever remains at the end is the removedChapters list that triggers an archive snapshot.

The lastReadChapterId is remapped through an alias table and dropped if it no longer matches a surviving chapter. epubStale is set true when new chapters were appended beyond the old range.

10Download Engine — in detail

See the Engines ▸ Download tab for the flow. Key details below.

Queue atomicity — the "R3" design

Queue read-modify-writes never use a coroutine Mutex or runBlocking (an ANR risk on the main thread). Instead everything serializes on the JVM monitor of the single AppStorage instance. Three primitives:

  • mutateQueueInPlace(transform) — atomic single-document RMW of download_queue.json
  • saveEnqueue(story, jobs) — atomic multi-document RMW (story + queue under one monitor acquisition)
  • recoverInterruptedDownloads() — re-promotes downloading → pending on cold start

The Activity's engine handles enqueue/control + UI callbacks; the service's engine runs the loop. Both share one AppStorage.

11EPUB Engine — in detail

See the Engines ▸ EPUB tab. Range selection & coverage warning via EpubSelection; filenames via EpubFilename (<sanitized>_Ch<start>-<end>.epub).

12Cleanup Engine — in detail

See the Engines ▸ Cleanup tab. ReDoS prevention: validateRegexRule rejects risky nested-quantified-group patterns and caps pattern length at 500 chars.

15Cross-Cutting Concerns

🔒 Concurrency & single-owner

  • One AppStorage per process; its monitor (@Synchronized) serializes all queue/story RMW — no coroutine mutex on the main thread
  • One TtsEngine on Dispatchers.Main.immediate + a Mutex around all state
  • CleanupEngine uses an AtomicReference swap — readers never lock
  • Per-host rate limiting via a ConcurrentHashMap of mutex+timestamp

💾 Durability & compatibility

  • Two primitives: DurableJson (AtomicFile) + AtomicFileWrites (temp+fsync+rename)
  • Uniform JSON envelope with legacy bare-payload read tolerance
  • Relative chapter paths; transactional staged-swap restore
  • Zip-slip protection + size limits on backup extraction
  • DownloadJobStatus.wire preserves legacy on-disk strings

⚙️ Background work — no WorkManager

Both background loops are foreground services with self-contained coroutine scopes and START_STICKY:
