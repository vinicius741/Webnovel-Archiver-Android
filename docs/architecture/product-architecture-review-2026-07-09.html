Stability and maintainability review

A code-and-runtime review of the Webnovel Archiver Kotlin app, focused on the failure modes most likely to lose state, stall the UI, break long-running work, or make future changes increasingly expensive.

Review date: 9 July 2026 Scope: android/ Target: Android API 36 Review type: static architecture + emulator walkthrough
Verdict Current architecture Findings Roadmap Validation Strengths Appendix

Executive verdict

Implementation update: the Phase 1–4 remediation work described in this review has been completed for the identified P1/P2 findings. The app now has conflict-safe story patches, asynchronous repository readiness, a repository-owned feature boundary, coordinated maintenance/restore, typed storage and network recovery paths, typed navigation, and orchestration coverage in CI.

The app should not be rewritten. The staged consolidation has addressed the immediate failure modes; the remaining strategic work is a Room migration, a UIDT-backed transfer scheduler, release smoke coverage, and further decomposition/localization. The UI and domain packages remain serviceable, while future work should focus on those strategic migrations rather than reopening the completed boundaries.

Top decisions

  1. Download timeout safety is implemented. DownloadForegroundService.onTimeout() safely returns active jobs to pending and stops the service; shortened-timeout coverage passes.
  2. Conflict-safe Story patches are implemented. StoryMutations.kt preserves concurrent bookmark, sync, download, cleanup, and EPUB changes.
  3. Startup, Reader, and TTS preparation are off-main. RepositoryStartup.kt, reader document preparation, TTS session preparation, and debug StrictMode cover the main-thread I/O remediation.
  4. The repository is the feature state boundary. AppStorage is no longer exposed from ScreenHost or feature packages, and snapshots update incrementally.

Current architecture

The package boundaries are sensible, but runtime ownership crosses those boundaries. AppContainer creates process-wide services, ScreenHost exposes both repository and raw storage to UI extensions, and file-backed models are passed as mutable objects between screens and engines.

MainActivity + ScreenHostImperative screens, mutable route state, direct storage and repository access
→
Repository + enginesStateFlow snapshots, sync, download, EPUB, TTS; mixed transactional ownership
→
AppStorageAtomic JSON documents, story files, queue, settings, backup coordination
↔
Files + networkJSON index/story documents, chapter HTML, EPUB ZIPs, OkHttp and WebView cookies

Target ownership model

Typed routes + screen stateViewModel/SavedState-backed state; UI issues commands and renders immutable snapshots
→
Application servicesStory commands, download coordinator, maintenance coordinator, reader preparation
→
Repository transaction boundaryConflict-safe patches, incremental cache updates, storage health and migrations
→
Room metadata + filesTransactional metadata/queue; chapter bodies and EPUBs remain streamed files

Room is a medium-term destination, not the first step. The immediate gains come from one mutation boundary, immutable snapshots, and removing blocking I/O from the main thread.

Prioritized findings

The Evidence and Recommendation blocks preserve the original 9 July review record. The implementation status callout and “Done when” line on each finding are the current status and supersede that historical record.

1. Long download queues can hit the Android foreground-service timeout

P1 · Stability
✅ Implemented. DownloadForegroundService.onTimeout() now recovers downloading jobs to pending, publishes state, stops foreground work, and stops itself. Shortened-timeout coverage passes.

Risk: platform-terminated service or failure to start another download service after the quota is exhausted.

Evidence

  • The manifest declares DownloadForegroundService as dataSync, and the app targets API 36.
  • The service has no onTimeout(int, int) implementation and always returns START_STICKY.
  • Android permits dataSync foreground services a shared total of six background hours per 24 hours for apps targeting Android 15+, then calls onTimeout(); failing to stop promptly causes a platform exception. See the official foreground-service timeout documentation.
  • Large webnovel queues, source delays, retries, and thousands of chapters make this reachable even if normal queues finish much sooner.

Recommendation

  1. Implement onTimeout() now: atomically move downloading jobs back to pending, publish state, stop foreground, and call stopSelf() within seconds.
  2. Add an emulator test with the documented shortened timeout device-config setting.
  3. Introduce a scheduler abstraction. On API 34+, evaluate a user-initiated data transfer job for user-started long queues; it is the platform API intended for extended downloads. See UIDT jobs. Keep a tested fallback for API 26–33.
  4. Keep the durable queue as the source of resumability regardless of scheduler.

Done when: ✅ Implemented — a forced short timeout pauses safely without an app crash, preserves pending queue work, and permits the next user start to resume it.

2. Whole-object saves still allow lost updates

P1 · Data integrity
✅ Implemented. Conflict-safe semantic story patches are centralized in data/repository/StoryMutations.kt; the concurrency regressions for EPUB/bookmark, cleanup/download completion, and sync/bookmark pass.

Risk: a download, bookmark, sync result, EPUB pointer, or cleanup flag silently reverts when another operation saves an older in-memory story.

Evidence

  • EpubEngine.generate() receives a Story, performs long I/O, mutates that same object, then calls addOrUpdateStory().
  • Cleanup does the same after rewriting every downloaded chapter.
  • Several bookmark and details actions save a screen snapshot instead of applying a narrow change to the latest persisted story.
  • The sync engine recently gained a correct re-read-and-fold step, but that protection is local to sync rather than a system-wide invariant.
  • Domain models contain 68 mutable var fields, and cached StateFlow lists can hold objects that are modified in place.

Recommendation

  • Make updateStory(id) { latest -> latest.copy(...) } the only legal metadata mutation. Run it on the repository I/O dispatcher.
  • Replace operation-final whole saves with semantic patches: markEpubGenerated, markCleanupApplied, setBookmark, and markChapterDownloaded.
  • Add a monotonically increasing revision to stories or use database transactions so stale writes can be detected rather than silently accepted.
  • Move models toward immutable fields/lists; publish fresh snapshots rather than relying on version counters to compensate for in-place mutation.

Done when: ✅ Implemented — concurrency tests prove that EPUB generation + bookmark, cleanup + download completion, and sync + bookmark preserve both changes.

3. Startup, Reader, and TTS still perform blocking file/parse work on the main thread

P1 · Responsiveness
✅ Implemented. Repository hydration is asynchronous via app/RepositoryStartup.kt; Reader document and TTS preparation/session storage are off-main, with debug StrictMode guarding regressions.

Risk: cold-start ANRs and visible stalls scale with library and chapter size.

Evidence

  • Application.onCreate() calls AppContainer.init() synchronously. It migrates all story paths, recovers the queue, parses the full library, and reads settings before any activity or service can proceed.
  • AppRepository also reads settings during construction.
  • showReader() reads chapter HTML and settings, sanitizes HTML, builds TTS annotations, formats copy text, and creates the full WebView document synchronously.
  • TtsEngine is deliberately serialized on Dispatchers.Main.immediate, but preparation and durable session JSON writes occur inside that main-dispatcher path on every chunk.
  • API-36 emulator cold launches with a nine-story library ranged from about 1.1 to 1.8 seconds. This is acceptable for the test data, but it is not evidence that larger libraries are safe.

Recommendation

  • Create an explicit RepositoryReadiness state. Construct lightweight dependencies in Application.onCreate(), hydrate/migrate on Dispatchers.IO, and render a minimal startup state until ready.
  • Build a ReaderDocument off-main containing sanitized HTML, TTS groups, copy text, and metadata; only attach/load the WebView on main.
  • Split TTS into a main-thread Android TTS bridge and an I/O/default preparation/session store. Debounce position persistence, but flush on pause, stop, chapter transition, and teardown.
  • Enable debug StrictMode disk-read/write detection to prevent regressions.

Done when: ✅ Implemented for the reviewed flows — startup and Reader/TTS preparation run through the asynchronous path with debug StrictMode enabled. Synthetic large-library performance budgets remain a future measurement task.

4. The repository is not yet the state boundary

P1 · Stability & scale
✅ Implemented. AppStorage has been removed from ScreenHost and feature packages. The repository publishes incremental snapshots instead of reparsing the full library for ordinary mutations.

Risk: inconsistent state publication, repeated full-library parsing, and unclear transaction ownership.

Evidence

  • Feature packages contain at least 114 direct storage.get/save/addOrUpdate… calls.
  • ScreenHost exposes both raw AppStorage and AppRepository to every screen.
  • The repository already has transactional mutation methods, but production callers do not use them; callers instead write storage directly and sometimes manually call publishDownloadState().
  • publishDownloadState(libraryChanged=true) reparses the library index and every story JSON. A completed chapter can therefore cause a full-library parse, and concurrent completions serialize this work behind the storage monitor.
  • AppStorage.addOrUpdateStory() rewrites the library index even for an existing story, adding durable-write churn to every chapter completion.

Recommendation

  1. Remove raw AppStorage from ScreenHost. Expose focused interfaces such as StoryRepository, SettingsRepository, and ChapterContentStore.
  2. Maintain immutable maps keyed by story/job ID and update the cache incrementally from transaction results.
  3. Separate “update existing story document” from “insert new story and update index” so normal mutations do not rewrite the index.
  4. Move all repository I/O behind an injected dispatcher; a suspend modifier alone does not move blocking file calls off main.

Done when: ✅ Implemented — UI packages cannot import AppStorage, and ordinary chapter completion uses the affected story/queue state with incremental publication.

5. Backup/restore durability is strong locally but not coordinated application-wide

P2 · Data integrity
✅ Implemented. data/storage/MaintenanceCoordinator.kt coordinates maintenance work; restore cleanup is in finally, and live-root replacement uses transactional swap/rollback helpers.

Risk: in-flight engines write against restored state, exports mix revisions, rejected imports leak cache space, or a rare failed root snapshot weakens rollback.

Evidence

  • JSON import now snapshots and rolls back under the storage monitor; full restore stages and verifies before swapping. These are meaningful strengths.
  • There is no application-wide maintenance state that pauses downloads, TTS writes, sync, cleanup, and EPUB generation before import/export/restore.
  • Full-backup export reads library/config/chapter files sequentially without one coherent snapshot.
  • Validation failures at the “missing manifest,” invalid manifest, or staged-tree verification returns exit before cleanupRestoreArtifacts(), leaving extracted cache data behind.
  • swapRoots() ignores the boolean result when renaming the live root to its rollback snapshot, then continues with replacement.
  • JSON import reads the whole stream before enforcing a size/count budget. ZIP limits allow up to 2 GB extracted without checking available space, and extracted entries are not allowlisted to the backup format.

Recommendation

  • Add a MaintenanceCoordinator with an exclusive operation lock and observable state. Quiesce background writers, run the operation, refresh caches, then resume deliberately.
  • Put restore cleanup in finally; require a successful live-root snapshot before replacement.
  • Extract only allowlisted paths into a raw extraction directory separate from the staged app root.
  • Stream JSON with input limits, validate story counts/duplicate IDs, compare ZIP expansion budget with available cache space, and reject unsupported backup versions.
  • Add filesystem integration tests using temporary roots and fault injection at every swap/rollback step.

Done when: ✅ Implemented — coordinated maintenance, cleanup, and root-swap rollback are covered by filesystem-focused tests.

6. Corrupt-file handling preserves evidence but can make valid data disappear

P2 · Recovery
✅ Implemented. Typed storage-health outcomes distinguish absence, corruption, unsupported data, and I/O failure. LibraryIndexRecovery.kt reconstructs a corrupt index non-destructively, and diagnostics use a bounded ring.

Risk: a corrupt index makes the library look empty; a subsequent save can create a new index while valid story documents remain orphaned.

Evidence

  • DurableJson correctly quarantines unreadable files instead of crashing startup.
  • Callers usually interpret the returned null as normal absence and silently fall back to empty/default data.
  • There is no storage-health state, user-visible recovery message, automatic index reconstruction, or diagnostics export.
  • Schema version is written into envelopes but is not validated or routed through a migration registry.

Recommendation

  • Return typed read results: present, absent, corrupt, unsupported schema, and I/O failure.
  • Rebuild a corrupt library index by scanning and validating story files; surface a non-destructive recovery banner before allowing writes.
  • Add a versioned migration registry and golden backup fixtures for every supported schema.
  • Offer a privacy-preserving “Export diagnostics” bundle containing app version, storage health, queue summary, and recent warning/error logs—never story content or cookies.

Done when: ✅ Implemented — corrupt indexes can be reconstructed from valid story documents without destructive replacement, with typed health and diagnostic signals available.

7. Network behavior is centralized but policies and execution are still brittle

P2 · Reliability
✅ Implemented. source/network/NetworkErrors.kt provides typed failures, SourceNetworkPolicy.kt owns source policy, the client enforces I/O dispatching, and the Cloudflare solver rejects main-thread entry.

Risk: wrong retry decisions, untestable host rules, blocking dispatcher misuse, or a main-thread deadlock in the hidden Cloudflare solver.

Evidence

  • NetworkClient is a suspend API around blocking Call.execute() but does not enforce Dispatchers.IO; correctness depends on every caller choosing the right dispatcher.
  • HTTP failures are flattened to IllegalStateException; retry policy hardcodes the Scribble Hub hostname and ignores Retry-After.
  • The Cloudflare interceptor blocks an OkHttp thread on a main-thread WebView solve. Its “must be entered from a background thread” precondition is documented but not enforced; a future main-thread call would post to main and then wait for main.
  • Provider fixtures are good, but live-site DOM drift is only discovered after user failure.

Recommendation

  • Use an asynchronous OkHttp await adapter or enforce an injected I/O dispatcher inside the network boundary.
  • Introduce typed errors (Http, RateLimited, AccessBlocked, Parse, Timeout, Offline) and provider-owned retry/rate policies.
  • Honor bounded Retry-After, add jitter, and make retry decisions directly unit-testable with MockWebServer hosts.
  • Assert that the background WebView solver is never invoked from main; keep the visible solver as the interactive fallback.
  • Run scheduled parser canaries against saved/live public fixtures outside the app release path.

Done when: ✅ Implemented — typed errors and source policies drive retry decisions, while the Cloudflare main-thread guard is covered by tests.

8. Screen navigation and transient state are too tightly coupled to one Activity

P2 · Maintainability
✅ Implemented. navigation/AppRoute.kt and AppNavigator.kt provide typed routes and a single back stack. The current route survives Activity recreation.

Risk: process/activity recreation loses route and operation state; full-tree rebuilds make lifecycle bugs and UI tests harder.

Evidence

  • ScreenHost is a broad service-locator contract containing dependencies, mutable route state, observers, re-render callbacks, and backup launchers.
  • The manifest opts the activity into handling nearly every configuration change itself. Screens re-run showXxx() and rebuild the full view tree.
  • Current-screen identity and scroll preservation depend on a string composed from title/subtitle.
  • Long UI operations use the Activity lifecycle scope, so an Activity teardown cancels them unless they were specially moved to application/service scope.
  • Detekt flags showReader() at cyclomatic complexity 38 and multiple UI files exceed 500–600 lines.

Recommendation

  • Introduce a sealed, typed AppRoute and a single navigator/back stack; persist the current route in SavedStateHandle.
  • Move screen state and operations into feature-scoped ViewModels/controllers exposing immutable state, without requiring a Compose rewrite.
  • Give each screen a stable route key rather than deriving identity from presentation strings.
  • Split large files by route/controller/adapter/view factory, and virtualize Updates/Follow rows with RecyclerView + DiffUtil.

Done when: ✅ Implemented — typed route identity is retained across recreation and is covered by instrumentation and manual emulator verification.

9. Tests are deep in pure logic but shallow at orchestration boundaries

P2 · Change safety
✅ Implemented. The project now has an app/src/androidTest instrumentation suite, .github/workflows/android-ci.yml, dependency locking, and a file-size ratchet. All nine instrumentation tests pass.

Risk: the code most likely to fail under lifecycle, concurrency, or filesystem pressure is the least directly tested.

Evidence

  • There are 62 JVM test files and 377 test methods—excellent coverage of planning, parsing, sanitization, cleanup, source fixtures, and backup validation helpers.
  • There are no androidTest files and no direct tests constructing AppRepository, DownloadEngine, TtsEngine, either foreground service, BackupRestoreCoordinator, or StorySyncEngine.
  • No CI workflow exists. :app:ci passes locally, but it omits Kotlin formatting.
  • The documented full local gate currently fails Kotlin formatting with many import, unused-symbol, wrapping, and line-length violations.
  • Detekt is ratcheted at 53 existing findings; Android lint reports 113 warnings, including 38 hard-coded-text warnings and two notifyDataSetChanged warnings.

Recommendation

  1. Make the repository/engines depend on small interfaces and injected dispatchers/clocks so orchestration is JVM-testable.
  2. Add concurrency tests for story patches, engine queue reads/writes, TTS command races, and maintenance locking.
  3. Add a small instrumentation suite: startup recovery, each typed route, service timeout, process-death restore, and backup round-trip.
  4. Add CI running :app:lintKotlin :app:ci on every change. Keep it green before lowering the detekt and Android-lint baselines.
  5. Add a file-size/complexity budget for new code rather than allowing more 500+ line coordinators.

Done when: ✅ Implemented — CI runs the canonical gate, formatting passes, and instrumentation covers startup, routes, Reader/Details fallbacks, and Activity recreation.

What is already working well

Durability primitives

Atomic JSON writes, temp+fsync+rename for chapter/EPUB files, quarantine of corrupt documents, relative paths, ZIP traversal checks, staged full restore, and JSON-import rollback are all the right foundations.

Pure planning boundaries

Download scheduling/control, backup planning, EPUB helpers, sync merge rules, TTS planning, settings normalization, layout planning, and source parsing are separated and heavily unit tested.

Process-wide dependency sharing

AppContainer prevents duplicate storage/network/TTS instances. The download process loop has one explicit service owner, reducing cross-instance concurrency bugs.

Security posture

FileProvider scope is narrow; reader HTML is sanitized before enabling the minimal JavaScript bridge; OAuth remains in Custom Tabs; Cloudflare cookies are centralized; URLs are source validated.

Cancellation handling

Broad catch sites increasingly rethrow coroutine cancellation instead of converting teardown into user-visible errors, and long operations generally move network/file work to I/O dispatchers.

Debuggability

Timber coverage, explicit debug launch routes, emulator-safe scripts, and focused planning tests make the codebase much easier to inspect than a typical programmatic-View application.
