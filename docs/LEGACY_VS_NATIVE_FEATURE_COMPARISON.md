# Legacy React Native vs. Native Kotlin — Feature Comparison & Gap Analysis

> **Scope:** A detailed, source-verified comparison of the legacy React Native / Expo app
> (`app/`, `src/`, `modules/`) against the current native Kotlin app (`android/`).
> Generated by reading the actual implementation files on both sides.
>
> **Date:** June 2026. The native Kotlin app is the active implementation; the React Native
> code remains in-repo for reference only.

---

## How to read this document

- ✅ **Parity** — feature exists on both sides with equivalent behavior.
- ➕ **Kotlin richer** — both have it, but the native app goes further.
- ⚠️ **Missing in Kotlin** — existed in React Native, **not** present in the native app (a real gap).
- 🔁 **Missing in RN** — exists only in the native app (informational; usually an improvement).

Findings are grouped by feature area. Each gap entry cites the legacy source location so it can be
re-implemented if desired.

---

## Executive summary

The native Kotlin app is a **near-complete rewrite** that reaches or exceeds the legacy RN feature
set in almost every area — downloads, EPUB generation, the cleanup rule system, backup/restore,
library management, and theming are all at parity or better. Data models are wire-compatible
(`Story`, `Chapter`, `EpubConfig`, `TtsSettings`, `RegexCleanupRule`, etc. match field-for-field),
which is why the JSON and full-ZIP backups interoperate between versions.

There are **four genuine functional gaps** where the legacy RN app offered something the native app
does not. All four are concentrated in two subsystems — **TTS media integration** and **the in-reader
TTS experience** — plus two smaller UX touches. They are summarized here and detailed below.

## 1. Sources & scraping — ✅ Parity

Both apps implement the same `SourceProvider` interface with the same two providers:

- **RoyalRoadProvider** — `royalroad.com`, story id `rr_<fiction#>`, chapter id from `/chapter/`.
- **ScribbleHubProvider** — `scribblehub.com`, AJAX-paginated TOC (requires a JS-enabled WebView).

`SourceRegistry`, `NovelMetadata`, `ChapterInfo`, and per-host rate limiting exist identically on
both sides. Neither version supports additional sites.

> The native app additionally centralizes URL import validation in `SourceUrlValidation`
> (rejects `/chapter/` URLs before sync), but this is a hardening improvement, not a feature gap.

---

## 2. Library — ✅ Parity (one minor UX gap)

Both apps offer the same library management:

- **Tabs** — create / rename / reorder / delete (deleted-tab stories move to Unassigned).
- **Search** — live text search over title + author.
- **Tag filtering** — content-genre tag chips with per-tag counts; the native app additionally
  renders **source-name** filter chips distinctly from genre tags.
- **Sort** — recency (default) / title / date added / last updated / total chapters / score, each
  ascending or descending. Identical option set on both sides (`useLibrary.ts` ↔ `LibraryQuery.kt`).
- **Selection mode** — multi-select with move-to-tab and delete.
- **Story cards** — cover (tap to zoom), title, author, status, progress, source badge, bookmark.
- **Add story** — by URL (with paste button) + browse-sources cards; both offer the same flow.

⚠️ ~~**Gap — swipe-between-tabs pager.**~~ ✅ **Closed.** The RN library wraps each tab's
story list in a `PagerView` (`react-native-pager-view`) so the user can **swipe horizontally to
switch tabs** (`app/index.tsx:186-209`, hook `useLibraryPager`). The native `LibraryScreen.kt`
now does the same when ≥2 tabs exist: a `ViewPager2` backed by `LibraryPagesAdapter` renders one
scrolling grid per tab, with two-way sync (swipe → updates the active tab indicator + persists;
tab-bar tap → animates the pager to that page). The common single-tab case keeps the original
single-grid layout unchanged.

---

## 3. Story details — ✅ Parity

The details screen matches field-for-field and action-for-action:

- Info panel: cover, title, author, status, score, tags, expandable description (+ copy), source URL.
- Action buttons: **Sync Chapters**, **Download All / Download Remaining (N)** (label adapts to
  remaining count), **Generate EPUB**, **Read EPUB**, **Regenerate** when stale.
- Overflow menu: Open Source, Select Chapters, Download Range, EPUB Settings, Apply Text Cleanup,
  Delete Novel.
- Chapter list: searchable; filter chips All / Downloaded / From Bookmark; per-chapter
  Read / Download / Mark-as-Read.
- Progress feedback: both render a progress bar during EPUB generation and download, with a status
  message. The native app even parses "Generating EPUB X/Y" messages into a determinate
  percentage (`StoryActions.kt:epubProgressFromMessage`).
- Archive badges + read-only guards on archived snapshots behave identically.

The chapter filter modes (`all` / `hideNonDownloaded` / `hideAboveBookmark`) and EPUB config
(`maxChaptersPerEpub` / `rangeStart` / `rangeEnd` / `startAfterBookmark`, clamped 10–1000) are
bit-for-bit identical models on both sides.

---

## 4. Download system — ✅ Parity

This is the most complex subsystem and it matches completely:

- Persistent on-device queue; queueing by all-chapters / explicit range / bookmark-forward /
  count-based / individual / multi-select.
- Global concurrency + delay, **and per-source overrides** (both apps expose per-provider
  concurrency/delay in settings — RN: `DownloadManagerSettingsModal` in the download manager;
  Kotlin: Settings → Source Overrides).
- Queue controls: pause/resume/cancel per job, per story, and globally; retry failed (per story
  and all); clear finished; remove terminal jobs.
- Retry with exponential backoff, error classification, retry countdown display.
- Foreground service with progress notification + pause/resume/stop actions.
- Startup recovery: interrupted `downloading` jobs → `pending`; unsupported-source jobs failed
  before consuming a worker slot; stuck story statuses recovered.

`DownloadJob`, status wire-strings, and `DownloadRangeMode` are wire-compatible across versions.

---

## 5. EPUB generation — ✅ Parity

Both use custom `java.util.zip`/ZIP+XML generation producing EPUB 2.0:

- Volume splitting by `maxChaptersPerEpub` (10–1000).
- Configurable chapter range + start-after-bookmark.
- Cover embedding with media-type detection, OPF metadata (title/author/description/tags/cover),
  NCX navigation.
- Stale-EPUB marking when the chapter list changes; multi-volume output via `epubPaths`.

`EpubConfig` and `EpubResult` models match exactly.

---

## 6. Text cleanup / sentence removal — ✅ Parity

Both apps implement the full cleanup system with identical models (`RegexCleanupRule` with
`appliesTo` of `download` / `tts` / `both`, `flags`, `enabled`):

- Sentence blocklist: add / edit-by-index / delete, with the legacy default anti-piracy phrases.
- Regex rules: add / edit, enable/disable toggle, delete, duplicate detection, validation.
- **Quick Separator builder**: characters + min repetitions + whole-line. ⚠️ *Note:* the native
  builder is actually **richer** here — it shows a **live pattern preview** (➕) that the RN
  `QuickBuilderForm` lacks.
- Export rules to JSON (both); neither imports rules.

⚠️ ~~**Gap — regex rule live sample-text preview.**~~ ✅ **Closed.** The RN `RuleDialog`
(`src/components/sentence-removal/RuleDialog.tsx`) included a **"test your rule" pane**: the user
pastes sample text into `previewInput` and the dialog runs the in-progress rule against it via
`applyTtsCleanupLines`, showing the cleaned output (`previewOutput`) in real time
(`RuleDialog.tsx:91-104`). The native `showRegexRuleDialog` (`CleanupScreen.kt`) now has the
equivalent — a **"Test Preview" pane** that runs `TextCleanup.previewRegexRule(pattern, flags,
sampleInput)` over pasted sample text, live-updating the output box on every keystroke in the
pattern/flags/sample fields. The native builder therefore matches the RN regex editor's preview
capability, on top of the live pattern preview it already had.

## 7. Backup & restore — ✅ Parity

Both apps offer the same two backup modes with the same manifest shapes:

- **JSON backup** — library metadata + tabs only (no chapters); merge-on-import.
- **Full ZIP backup** — settings + tabs + library + chapter HTML + cleanup rules + TTS settings +
  display prefs; transactional restore with pre-restore validation.
- Cleanup-rules export to JSON (both); neither imports cleanup rules.

`BackupExportPlanning`/`BackupMergePlanning`/`FullBackupRestorePlanning` in the native app keep the
same on-disk format and merge policy as the RN `BackupService`, which is why backups move between
the two apps. Neither version has scheduled/automatic backups — all manual.

## 8. Text-to-Speech (TTS) — ✅ Parity (all gaps closed)

The core TTS engine matches: paragraph-oriented chunking by `chunkSize`, configurable pitch /
rate / voice, pause/resume/skip, session persistence, auto-resume on restart, and a foreground
service with notification action buttons (Prev / Play-Pause / Next / Stop). `TtsSettings` and
`TtsSession` are wire-compatible.

The **OS-integration layer and in-reader experience previously diverged**, but all four gaps are now
closed. The TTS engine is a process-wide singleton (`AppContainer.ttsEngine`) shared by the activity
reader and the foreground service, with a multicast `onStateChanged` snapshot
(`TtsPlaybackSnapshot`) so the MediaSession/notification (service) and the highlight/transport
(reader) stay in lockstep.

### ✅ Gap 1 — `MediaSession` / media-style notification (was High — closed)

The legacy app shipped a dedicated **Expo native module**, `modules/tts-media-session/`, whose
Kotlin side (`TtsMediaSessionService.kt`) owns a real Android **`MediaSession`** and publishes a
**`MediaStyle` notification**.

The native Kotlin `TtsForegroundService.kt` now does the same: it creates a `MediaSessionCompat`
whose callback maps transport controls onto the engine, and `buildNotification` swaps the old
`BigTextStyle` for `MediaStyle(...).setShowActionsInCompactView(0,1,2)`. On every state change the
service refreshes the session's `PlaybackStateCompat` (PLAYING/PAUSED/STOPPED) and
`MediaMetadataCompat` (title / "Webnovel Archiver" / track number = chunk index / total chunks).
Consequences now delivered:

- **Lock-screen / system-UI media controls** appear (the standard Android media playback card on the
  lock screen and in the quick-settings shade).
- **Metadata in the system media UI** (title, chunk position).

### ✅ Gap 2 — Headset / Bluetooth media-button support (was High — closed)

The service's `MediaSession` carries a `MediaButtonReceiver.buildMediaButtonPendingIntent(...)` media-
button receiver, the manifest registers `androidx.media.session.MediaButtonReceiver` for
`ACTION_MEDIA_BUTTON`, and the session callback's `onMediaButtonEvent` maps the headset hook /
Bluetooth play-pause / media keys onto the engine. A **single headset/Bluetooth play-pause click now
starts/pauses TTS**.

### ✅ Gap 3 — In-reader TTS highlighting + tap-to-start (was Medium — closed)

The reader now produces **TTS-aware HTML** via `TextCleanup.prepareTtsAnnotatedHtml` — a port of the
legacy `prepareTTSContent` (`src/utils/htmlUtils.ts:33-123`) that walks the chapter's block elements,
groups them into `chunkSize`-sized groups, and tags each with `data-tts-group="<index>"` + a
`tts-chunk` class. The chunk list it returns is **byte-for-byte aligned** with
`TextCleanup.prepareTtsChunks` (the engine's chunk list), so a chunk index the engine reports maps
1:1 to the highlighted group.

The reader CSS now ships the `.tts-active` highlight (translucent accent background + solid accent
left border), and an injected `WnaTts` script (a) exposes `WnaTts.setActive(i)` which the host calls
via `evaluateJavascript` as the live `TtsSession.currentChunkIndex` advances (no page re-render, no
flash), and (b) lets the reader **double-tap (or double-tap) a paragraph to start TTS from that
paragraph** via a single-method `@JavascriptInterface onTtsStart(index)` → `engine.playFromChunk(...)`.

**Security note:** because this requires JavaScript in the reader WebView, chapter HTML is now first
sanitized through `ChapterHtmlSanitizer` (a strict Jsoup `Safelist` that strips scripts, `on*`
handlers, `<iframe>`/`<object>`, and `javascript:` links) before rendering. File/content access stays
off and Safe Browsing stays on, so the only script that runs is the one this app injects — a net
hardening over the prior JS-off-but-unsanitized render.

### ✅ Gap 4 — Floating in-reader TTS transport (was Medium — closed)

The reader now shows a **persistent floating TTS transport** docked above the chapter nav
(`readerTtsTransport` in `ReaderScreen.kt`), mirroring the legacy `TTSController.tsx`: it displays
`"Chunk X / Y"` progress (or "Buffering"), offers skip-previous / play-pause / skip-next / stop, and
is refreshed live from `ttsEngine.onStateChanged`. It appears only while a TTS session exists for the
chapter currently open. The existing modal `showReaderTtsPanel` (the "Read aloud" app-bar action)
remains as the discoverable entry point.

### ➕ TTS voice picker — Kotlin richer

One area where the native app is *thinner* in presentation but equivalent in function: the RN
`TTSSettingsModal` shows a searchable voice list with **quality badges** ("Enhanced / High Quality",
`TTSSettingsModal.tsx:110-113`) and per-voice locale. The native voice picker
(`showTtsVoicePicker`) lists "name (locale)" without quality sorting/badges. Functionally the same
voices are selectable; only the presentation differs.

## 9. Reader — ✅ Parity

The reader matches:

- WebView-based chapter rendering with sanitized HTML, responsive CSS, `max-width:100%` images.
- Prev/Next chapter navigation with a **"X / total" position indicator** (both the subtitle and the
  bottom nav show it).
- **Reader settings panel**: font-size stepper (A− / percentage / A+), dark-reader toggle, copy
  chapter text. (The RN equivalent exposes font family + line-height via theme typography rather
  than a user stepper; the native stepper is arguably more discoverable.)
- Bookmark / mark-as-read toggle persisted as `lastReadChapterId`.

---

## 10. Browser / import — ✅ Parity

Both ship an in-app WebView importer with the same controls: address bar with https lock icon,
Go / Forward / Refresh, progress bar, "Save imported novel to" tab selector, Import, and Open in
external browser. Both resolve non-URL input to a **Google search** and both block Google OAuth
URLs from in-app navigation. `BrowserUrlPlanning.kt` mirrors RN's `resolveUrl`/`isGoogleAuthUrl`
exactly.

---

## 11. Settings & theming — ✅ Parity

The native settings hub is a **superset** of the RN one:

- **Appearance** — theme picker (Obsidian / Midnight / Forest / Classic Light), large-screen layout
  mode, EPUB volume folding mode.
- **Downloads** — concurrency, delay, max chapters per EPUB, **per-source overrides**
  (the RN screen surfaces concurrency/delay/source-overrides inside the *download manager* modal
  rather than settings; both expose them, just in different places).
- **TTS** — pitch / rate / chunk size / voice picker + saved-session resume/clear.
- **Library** — manage tabs.
- **Data** — text cleanup rules, clear local storage.
- **Backup** — JSON export/import, full ZIP export/restore.

Four themes exist on both sides; the native `AppTheme` carries equivalent color/shape/elevation
data to RN's `AppTheme` (the RN theme also defines `readerFontFamily`/`readerLineHeight` typography
fields, which the native reader handles via its own renderer CSS + the font-scale stepper rather
than per-theme font metadata — same end result, different mechanism).
