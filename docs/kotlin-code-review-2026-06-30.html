00Executive summary

The headline: this is a deliberately designed, well-documented codebase — not a naive one. The hardest problems (single source of truth, WebView lifecycle, foldable handling, atomic file writes) have clearly been reasoned about. But there are a handful of real correctness and data-loss risks, and a long tail of maintainability improvements.

The three things to fix first. Everything else can wait.
  1. Zero logging anywhere. Every catch swallows exceptions into a toast — field debugging is effectively impossible. (See Observability.)
  2. catch(Throwable) swallows CancellationException in 5 suspend/worker paths, while restore catches Throwable too. (See Error handling.)

03Persistence & data layer

The data layer is built around AppStorage (an 872-line god-object) doing JSON-file persistence with atomic writes and a versioned envelope, wrapped by AppRepository which exposes StateFlows and serializes every mutation with synchronized(storage){…}. The atomic-write discipline is good in intent but has real correctness holes.

MediumPersistence · durability data/storage/DurableJson.kt:44-56

P1 · Atomic JSON write never calls failWrite on failure

Now
The write uses out.use { … atomic.finishWrite(it) }. If gson.toJson() or out.write() throws (disk full, OOM), use{} calls close() — not failWrite(). The previous version of this report overstated this as "commits a half-written live file": Android's AtomicFile usually preserves/restores the previous base file on the next read when a backup/new file exists.
Risk
The rollback contract is still incomplete. Failed writes can leave stale .new/.bak artifacts, first-time writes have no previous document to restore, and callers get no distinct corruption signal. Combined with P2, any partial first-write or unreadable document is still collapsed to "missing" and silently hidden.
Fix
Wrap in try { … finishWrite(out) } catch (e) { atomic.failWrite(out); throw e }. Drop the use{} around the whole thing (finishWrite/failWrite already close).
HighError handling · data-loss data/storage/DurableJson.kt:65-73 → AppStorage.kt:82,838

P2 · JSON parse failures silently delete user data

Now
readAtomic wraps both the file read and every gson.fromJson in runCatching{}.getOrNull(). Downstream, getLibrary() uses mapNotNull and getQueue() returns mutableListOf() on null.
Risk
A single corrupt story JSON makes that story silently vanish from the library. A corrupt queue makes the whole download queue disappear. Combined with missing diagnostics and incomplete write-failure cleanup (P1), storage problems become invisible — no log, no error, no recovery path. This is the single highest-impact correctness risk.
Fix
Distinguish "file missing" (null is fine) from "parse failed" (throw, or return a sealed ReadResult.Corrupt). At minimum log parse failures and quarantine the bad file (rename to .corrupt) so data isn't hidden and can be recovered.
HighPersistence · atomicity data/storage/AppStorage.kt:641-656

P3 · "Atomic swap" restore is neither atomic nor rolled back on failure

Now
swapRoots tries renameTo; on failure it falls back to a non-atomic recursive copy. On any exception during the restore path, the outer catch(Throwable) returns an error string and the preRestoreSnapshotDir with the old data is never restored.
Risk
If the cross-filesystem copy throws midway, root is left as a half-populated tree, the snapshot still holds the good data, but nothing puts it back. The user ends up with a broken library after what they were told was a safe restore.
Fix
In the catch path, restore from preRestoreSnapshotDir back to root. Don't delete the snapshot until root is verified post-swap.
MediumPersistence · durability data/storage/AtomicFileWrites.kt:78-96

P4 · fsync on the temp file but no directory fsync + destination pre-deleted

Now
Before rename, the temp file is fsync'd, but the parent directory is not. Worse, renameOnto calls destination.delete() before renameTo(), removing the durability guarantee — between delete and rename there is no file at the destination.
Risk
Crash safety on ext4/f2fs requires fsync'ing the parent dir after the rename, otherwise the rename itself may not survive power loss. Pre-deleting the destination means a crash loses both the old and new content — directly contradicting the class doc ("the previous content survives until the new file is complete").
Fix
Don't pre-delete. Use a primitive that provides checked atomic replace semantics on Android/Linux (or switch these writes to AtomicFile too), and fsync the parent directory after the rename/replace. If a fallback copy is needed, keep an explicit backup until the replacement has been verified.
MediumPersistence · migration data/storage/DurableJson.kt:21,59-75

P5 · Schema version is written but never read or migrated

Now
Envelope.schemaVersion = 1 is persisted on every write, but readAtomic parses the envelope and never reads schemaVersion.
Risk
The doc claims the envelope exists "so future migrations can detect the on-disk shape," but there is no migration hook, no version check, and nowhere to plug one in. The moment a field shape changes, old files silently fail to parse → P2 swallows them.
Fix
Add a schemaVersion check in readAtomic routing to a migration registry; add a test that a v1 file still loads after bumping CURRENT_SCHEMA_VERSION.
MediumPersistence · correctness data/storage/DurableJson.kt:67-74

P6 · Envelope detection is a string-contains heuristic with a lossy double round-trip

Now
if (text.contains("\"payload\"")) decides envelope vs legacy. Then Envelope.payload: Any? is parsed as a Gson LinkedTreeMap, re-serialized with gson.toJson(it) and re-parsed into T.
Risk
(a) A legacy object with any top-level or nested "payload" key misdetects as an envelope. Plain user text is less likely to trigger this because quotes are JSON-escaped. (b) The double round-trip is type-erasing: unknown numbers become Gson Doubles before being coerced back into typed models. Current epoch-millisecond timestamps are still exactly representable, but the pattern is fragile and repeats in importBackupUri and restoreFromZip.
Fix
Parse once into a typed Envelope<T> via a parameterized TypeToken; detect shape by a top-level "schemaVersion" key.
MediumConcurrency data/storage/AppStorage.kt:78-170

P7 · Per-method @Synchronized is inconsistent — settings writes are unlocked

Now
Library/queue/story mutations are @Synchronized, but saveSettings, saveSourceDownloadSettings, saveChapterFilterSettings, saveDisplayPreferences, saveTabs, saveTtsSettings, saveTtsSession go straight to write() unsynchronized.
Risk
The "one lock around every multi-step mutation" promise is false. AppRepository.saveSettings also does storage.saveSettings + _settingsFlow.value = settings non-atomically.
Fix
Either make all mutators @Synchronized, or remove per-method locks and centralize locking in AppRepository exclusively.
MediumConcurrency · main-thread I/O data/storage/AppStorage.kt:269-272 → AppRepository.kt:142

P8 · readChapter does synchronous disk I/O but is not a suspend function

Now
readChapter does File(it).takeIf(File::exists)?.readText() on the calling thread, and AppRepository.readChapter is a plain (non-suspending) passthrough.
Risk
Chapter HTML can be tens-to-hundreds of KB. Any UI caller that forgets withContext(Dispatchers.IO) blocks the main thread. The API gives no signal that it's blocking — it relies on caller discipline.
Fix
Make it suspend and read on Dispatchers.IO internally, or return a Flow.
MediumAPI design data/repository/AppRepository.kt:38-40

P9 · Repository exposes its AppStorage publicly, breaking single-source-of-truth

Now
val storage: AppStorage is public. DownloadEngine immediately does private val storage = repository.storage and calls storage.saveChapter, storage.mutateQueueInPlace, storage.getQueue directly — bypassing the repository and its flow bookkeeping.
Risk
Two write paths into the store. Engines must remember to call repository.publishDownloadState(...) manually after every direct write (by convention, not enforced). This is the root cause of needing the downloadStateVersion counter — a symptom of the underlying dual-writer problem.
Fix
Make storage private. Give engines the narrow operations they need as repository methods, so every write inherently refreshes the flows.
MediumAPI design · god object data/storage/AppStorage.kt (entire file, 872 lines)

P10 · AppStorage is a god object

Now
One class owns: JSON envelope read/write, library index + per-story files, chapter HTML I/O, EPUB binary streaming, settings/config/tabs/cleanup/TTS persistence, JSON backup export/import + merge, full ZIP backup + transactional restore with staging + zip-bomb protection, legacy path migration.
Risk
Mixing risky backup/restore with day-to-day persistence in the same class makes both hard to test, and every @Synchronized lock covers all of them (a slow restore blocks settings reads).
Fix
Split into AppStorage (JSON config + library + queue), ChapterStore/EpubStore (binary blobs), BackupExporter/BackupImporter (JSON + ZIP + restore staging), each with its own locking.
LowAPI design data/repository/AppRepository.kt:41-64

P11 · Four overlapping StateFlows for the same underlying state

Now
_libraryFlow, _queueFlow, _downloadState (a snapshot of both), and _downloadStateVersion (a counter) are all maintained in publishDownloadStateLocked.
Risk
Redundant observable surfaces for one logical state; observers get inconsistent "what changed" semantics. downloadStateVersion is acknowledged as legacy in its own doc but still updated.
Fix
Expose one StateFlow<DownloadUiSnapshot> (or two: libraryFlow + queueFlow). Delete downloadStateVersion after migrating observers.
LowPersistence · leak data/storage/AtomicFileWrites.kt:72-88

P12 · Temp files leak on write failure

Now
tempSibling creates ${name}.tmp.$n; if block/writeBytes throws before renameOnto, the temp is never deleted.
Risk
Orphaned .tmp.N files accumulate in novels/ and epubs/ over a long session with intermittent disk errors (one per chapter save).
Fix
try { … } finally { if (!renamed && temp.exists()) temp.delete() }.
LowDocumentation drift data/storage/AppStorage.kt:219-220,236 · sync/StorySyncEngine.kt:14-16

P13 · Stale comments reference removed designs (txMutex, core package)

Now
Comments explain why mutateQueueInPlace exists "instead of the repository's txMutex path" — but there is no txMutex anywhere. StorySyncEngine says "Lives in the core package" but the package is …sync.

Comments describing a removed design actively mislead future readers about how locking works today.

04Coroutines & concurrency

The discipline here is mostly good: no GlobalScope, no executable runBlocking, no Thread.sleep in production, and StateFlows are exposed read-only via asStateFlow(). The problems are about thread-blocking locks inside coroutines, lifecycle-ungated collection, and one TTS engine whose documented mutex discipline is not actually applied to public controls.

Verified clean. grep confirms: 0 GlobalScope, 0 executable production runBlocking (comment/KDoc references only), 0 production Thread.sleep. MutableStateFlow instances are all private with public read-only StateFlow views. Scopes are tied to lifecycles (lifecycleScope in the activity, SupervisorJob() in engines). This is better hygiene than many production apps.
HighConcurrency · structured concurrency data/repository/AppRepository.kt:88,178-228 · download/DownloadEngine.kt:178,235,272

C1 · synchronized(storage) inside suspend functions blocks the IO dispatcher

Now
Every repository mutation is suspend fun … = synchronized(storage) { storage.*(); publishDownloadStateLocked(...) }. The download processLoop holds this same monitor across getQueue() + selectEligibleJobs() + saveQueue().
Risk
This pins the calling coroutine's thread (typically a Dispatchers.IO thread, 64-thread cap) on a JVM monitor while doing file I/O. A repository.upsertStory() from a UI coroutine blocks until the download loop releases. Under a few concurrent enqueue/sync calls this is a contention bottleneck; with one fat library save it stalls the whole pipeline. Mixing synchronized (thread-blocking) with coroutines defeats structured concurrency.
Fix
Replace the shared JVM monitor with a single kotlinx.coroutines.sync.Mutex held only across the critical section, or route mutations through a single-writer actor/channel on Dispatchers.IO. Keep file I/O inside withContext(Dispatchers.IO).
MediumLifecycle feature/library/LibraryScreen.kt:283 · feature/details/DetailsScreen.kt:472 · feature/downloads/QueueScreen.kt:112

C2 · Flows collected on the activity scope with no repeatOnLifecycle

Now
Every screen that observes repository.downloadState launches the collector on MainActivity.scope (wrapping lifecycleScope). It does not gate on STARTED/RESUMED. Cancellation relies on screen() overwriting screenObserver and onDestroy.
Risk
Collection continues while the activity is STOPPED (another app in foreground, or this app's own foreground service running), so runOnUiThread { refresh(...) } can mutate a detached view tree. The manifest declares all configChanges so the activity is never recreated — which masks lifecycle bugs that would bite on a stock activity.
Fix
Use lifecycleScope.launch { repeatOnLifecycle(Lifecycle.STARTED) { repository.downloadState.collect { … } } } so collection pauses when the UI isn't visible. Lower priority than usual given the no-recreate design, but it's the correct View-system idiom.
MediumConcurrency · TOCTOU download/DownloadEngine.kt:348-350

C3 · isCancelled/isPaused read the queue outside any lock

Now
storage.getQueue().any { it.id == id && it.status == … } with no synchronized(storage), used to guard whether to mutate a job after fetch.
Risk
A status change racing between read and the subsequent updateJob/handleJobError can mark a just-cancelled job as Completed/Failed. Benign in practice (eventually-consistent UI) but it's a TOCTOU that contradicts the otherwise strict "every queue RMW under the monitor" rule.
Fix
Do the status check inside mutateQueueInPlace.
LowConcurrency · latent download/DownloadEngine.kt:43,188,205,228

C4 · nextAllowedJobAtBySource is a plain MutableMap mutated off-monitor

Now
A mutableMapOf<String, Long>() read in selectEligibleJobs/nextWakeUpAt and written after each batch, all within the single processLoop coroutine.
Risk
Safe today only because the loop is single-threaded and processJob jobs don't touch it. But "only one caller ever" is exactly the invariant that breaks silently the moment a control method or a second start() touches it.
Fix
Make it ConcurrentHashMap, or move it inside the locked section with a clear ownership contract.
LowConcurrency · scope subtlety app/MainActivity.kt:62

C5 · CoroutineScope(lifecycleScope.coroutineContext) copies the Job — a subtle second scope

Now
override val scope by lazy { CoroutineScope(lifecycleScope.coroutineContext) }. This captures the same Job/CoroutineContext into a new scope wrapper.
Risk
Cancellation still works because the Job instance is shared, but it's a non-obvious pattern — a reader can't tell at a glance whether this is a child scope or an independent one.
Fix
Just use lifecycleScope directly (it already is a LifecycleCoroutineScope), or document why the wrapper exists.
MediumTTS · thread safety tts/TtsEngine.kt:31-43,97-186,241-263

C6 · TtsEngine's mutex contract is documented but not enforced

Now
The class KDoc says public controls and TTS callbacks route through stateMutex on the engine scope. In reality, stateMutex.withLock is used only during onInit and onDone. Public methods such as play, playFromChunk, pause, stop, next, and resumePersistedSession mutate chunks, index, session, and playbackActive directly.
Risk
Notification actions, reader buttons, MediaSession callbacks, and TTS progress callbacks can interleave state changes. The same methods also perform file-backed storage reads and text cleanup on the caller thread, often the main/service thread. That can produce stale snapshots, skipped chunks, duplicated listener updates, and UI/notification jank on large chapters.
Fix
Make public controls dispatch onto the engine scope and wrap all state reads/writes in stateMutex.withLock. Move storage reads and chunk preparation to an IO/default dispatcher, then publish snapshots back on main.
MediumForeground service · lifecycle tts/TtsForegroundService.kt:432-447,69-92,288-291

C7 · Cold TTS commands can start a foreground service without foregrounding it

Now
TtsForegroundService.command(...) calls startForegroundService on Android O+ for every action, including PAUSE, NEXT, PREVIOUS, and STOP. Several of those onStartCommand branches do not call startForegroundIfNeeded. The MediaSession onStop path also calls stopForeground directly instead of the helper that resets foregroundStarted.
Risk
If the service is cold-started by a reader/settings command that is not START or RESUME_SESSION, Android expects startForeground() quickly and can throw a foreground-service start-timeout exception. The direct onStop path can also leave foregroundStarted = true in the same service instance, making the next start skip startForeground().
Fix
Only use startForegroundService for actions that will enter foreground immediately, or have every cold command call startForegroundIfNeeded before doing work. Route all stop paths through stopForegroundAndReset().

05Error handling

No printStackTrace(), no crashy ad hoc exception dumps — good instincts. But the consistent catch(Throwable) + toast/string pattern eats CancellationException in suspend/worker paths, restore catches Throwable too, and detekt's rule that would catch this is disabled project-wide.

HighError handling · structured concurrency feature/story/StoryActions.kt:125,179,262,321

E1 · catch(Throwable) swallows CancellationException in 4 suspend sites

Now
Four suspend-adjacent call sites wrap work in try { … } catch (error: Throwable) { onError(error) / toast(error.message) / clearStoryOperation(...) } and never rethrow CancellationException. Together with the download worker (E2), that is five suspend/worker cancellation paths; restore is the sixth production catch(Throwable) but is not a suspend function.
Risk
This is the canonical Kotlin-coroutines footgun. When the parent scope is cancelled (Activity destroyed, lifecycleScope cancelled, job paused), the cancellation signal is intercepted and converted into a user-facing toast/`onError` instead of propagating. This leaves coroutines running after their scope is dead, leaks work, and produces spurious "failed" toasts on back-navigation/config changes.
Fix
In every catch(Throwable) inside a suspend context: 
} catch (error: Throwable) {
    if (error is CancellationException) throw error
    // ... existing handling
}
Use kotlinx.coroutines.NonCancellable where cleanup genuinely must run after cancellation.
HighError handling · structured concurrency download/DownloadEngine.kt:290-293

E2 · Download worker only handles cancellation for the paused case

Now
} catch (error: Throwable) { if (error is CancellationException && isPaused(job.id)) return; if (!isCancelled(job.id)) handleJobError(job, error) }
Risk
A genuine scope cancellation that is not a user pause falls through to handleJobError, which mutates the queue and emits progress — running side-effects on a coroutine that should already be dead, racing with teardown. It also catches OutOfMemoryError/StackOverflowError as ordinary job failures.
Fix
catch (e: CancellationException) { …; throw e } first, then catch (e: Exception) { if (!isCancelled(job.id)) handleJobError(...) }. Don't catch Throwable — catch Exception.
MediumError handling · diagnostics data/storage/AppStorage.kt:558-561

E3 · Broad catch(Throwable) for restore flattens all failures to a string

Now
The entire restore flow is wrapped in catch (error: Throwable) returning "Restore failed: ${error.message …}".
Risk
Swallows CancellationException/InterruptedException; OutOfMemoryError on a huge ZIP, ZipException on a corrupt archive, and IOException on disk-full all surface as the same message, and none is logged (see Observability). The verifyStagedTree guard is commendable.
Fix
Catch IOException + JsonParseException + ZipException specifically; let CancellationException/Error propagate. Return a sealed RestoreResult so the caller can decide presentation.
MediumTooling · lint app/detekt.yml:54-55

E4 · detekt TooGenericExceptionCaught is disabled project-wide

Now
The rule that catches catch(Throwable)/catch(Exception) is blanket-off "because download/TTS error classification intentionally catches Throwable."
Risk
That rule is exactly what would have surfaced E1/E2/E3. Disabling it project-wide to accommodate a few legitimate classifier sites lets many latent bugs through.
Fix
Re-enable the rule and suppress it with @Suppress("TooGenericExceptionCaught") only at legitimate classifier/bridge sites. Note that DownloadErrorClassifier itself does not catch Throwable; the broad catches are in callers such as DownloadEngine.processJob and restore. Converts a global blind spot into an explicit per-site opt-in.
LowError handling · API data/storage/AppStorage.kt:564-602

E5 · extractZipWithLimits return value ignored; error() used for control flow

Now
Returns the entry count; the caller ignores it. Limit violations throw IllegalStateException via error(), caught by the outer catch(Throwable).
Risk
Dead return value; inconsistent with the String?-returning validators (BackupExportPlanning, JsonBackupValidation).
Fix
Return a sealed result or String? error like the other validators.

06UI / View layer

A single-Activity app hosting a FrameLayout, swapping entire view trees per screen via internal fun ScreenHost.showXxx() extensions — no Fragments, no ViewModels, no XML layouts, everything built programmatically. This is intentionally NOT Compose. The findings judge it on View-system merits; none warrant a migration.

HighRecyclerView · performance feature/details/ChapterListAdapter.kt:95-110,145

U1 · notifyDataSetChanged() + full row rebuild on a list that can hold thousands of items

Now
update(...) reassigns fields then calls notifyDataSetChanged(). onBindViewHolder does row.removeAllViews() and re-builds the entire row (status slot, title column, subtitle, bookmark icon) on every rebind. The Lint NotifyDataSetChanged warning is already firing.
Risk
For novels with hundreds/thousands of chapters, every download-progress tick or filter keystroke causes a full rebind with no DiffUtil, no payload, no stable ids — and because rows are fully rebuilt, recycling only saves the top-level LinearLayout. This is an outlier: QueueGroupAdapter and ChapterSelectionAdapter do use DiffUtil correctly elsewhere in the same app.
Fix
Set setHasStableIds(true) + getItemId keyed by chapter.id; switch to DiffUtil.calculateDiff or ListAdapter; inflate the static row structure once in onCreateViewHolder and mutate only status/title/icon in bind. For progress-only updates, use notifyItemChanged(pos, payload) with a payload-aware onBindViewHolder overload.
HighRecyclerView · performance feature/downloads/QueueScreen.kt:192-198 → addStoryGroup:312-376

U2 · Queue adapter rebuilds an entire expandable card subtree on every bind

Now
Each onBindViewHolder calls holder.container.removeAllViews() then addStoryGroup(...), which constructs a fresh card with header rows, progress summary, and one child row per job. The recycled FrameLayout is reused but the entire content tree is re-created from zero.
Risk
Effectively "LinearLayout inside a RecyclerView" — recycling only saves the outer FrameLayout; the heavy card + all chapter job rows are redone every bind. Scales poorly with DiffUtil updates. Same pattern in DetailsHeaderAdapter.onBindViewHolder.
Fix
Make the group ViewHolder hold the card structure (header + nested RecyclerView or LinearLayout body) and update only contents; or use a ConcatAdapter so each job row is a real recycled item.
MediumState · process death navigation/ScreenHost.kt:71-86 · app/MainActivity.kt:70-73,190-193

U3 · No ViewModel / SavedStateHandle — nothing survives process death

Now
Because configChanges is declared, rotation/fold/theme change never recreate the activity; onConfigurationChanged calls rerender?.invoke() to rebuild the view tree. All transient state lives on ScreenHost/MainActivity fields.
Risk
This only works because config changes are suppressed. The OS can still reclaim the process (low memory, background too long); on return the activity is recreated with no onSaveInstanceState/SavedStateHandle — activeStory, storyOperation, storyExpandOverride, scroll position, and addStoryScreenState are all lost, dropping the user back to the Library root.
Fix
Persist the minimal navigation state (active story id, current screen) via onSaveInstanceState, or introduce lightweight ViewModels for at least the reader/details screen.
MediumArchitecture · dead abstraction feature/library/LibraryPresenter.kt:41-90

U4 · LibraryPresenter exists but is unused — the screen re-derives state inline

Now
LibraryPresenter is documented as the "ViewModel/presenter" owning state derivation. But showLibrary() never instantiates it — it reads storage.getLibrary() directly and calls LibraryQuery.filterAndSort(...) inline on every filter keystroke. LibraryUiState/LibraryStorySummary are never referenced outside the presenter file.
Risk
Dead abstraction giving false confidence. The documented benefit ("disk reads off the render path") is not realized: storage.getLibrary() is a JSON read on the main thread during render, and the cached repository.library() is bypassed.
Fix
Wire LibraryPresenter in, or delete it. At minimum, replace storage.getLibrary() with repository.library().
MediumTheming · global mutable state ui/AppTheme.kt:246-256

U5 · ThemeManager is a process-global singleton; theme changes aren't reactive

Now
object ThemeManager { @Volatile var current: AppTheme }. Every view reads ThemeManager.colors at construction time and bakes the resolved Int into its drawable / text color.
Risk
(a) Not reactive — picking a new theme in Settings works only because it triggers a full re-render; a view patched in place keeps old colors until rebuilt. (b) No system dark-mode mapping via resource qualifiers — uiMode is in configChanges so the system dark toggle does nothing unless it maps to a stored theme id.
Fix
Expose a themeId: StateFlow<String> and have screens re-render on change (the infra already exists), or drive theme through ContextThemeWrapper + Material ColorScheme so views resolve colors lazily. Document explicitly that dark mode is manual-only.
MediumAccessibility ui/HostUi.kt:182-195 · feature/library/LibraryStoryViews.kt:193-201 · feature/details/ChapterListAdapter.kt:189

U6 · Status conveyed by color/dot shape is invisible to TalkBack

Now
Icon buttons and the bookmark icon do have contentDescription (good). But the status dot indicator, the archived badge ImageView, and the chapter download-state dot/spinner are unlabeled. ZoomableImageView has no role/label for the pinch-zoom gesture.
Risk
Status conveyed purely by color/dot shape is invisible to TalkBack users. The chapter row's subtitle text partially compensates, but the library card's progress bar and archived icon don't.
Fix
Add contentDescription (or importantForAccessibility = NO + adjacent label) to status dots/badges. Mark purely decorative ImageViews importantForAccessibility = NO.
MediumWebView · coupling feature/reader/ReaderScreen.kt:86-111,182-217

U7 · Reader WebView rebuilt per render; TTS bridge does disk I/O on the UI thread

Now
Each showReader() creates a new WebView + new ReaderTtsBridge JS interface. The reader's rerender is a full showReader(), so fold/rotate destroys and reconstructs the WebView. The onTtsStart JS callback runs on the WebView's internal thread and calls app.runOnUiThread { … storage.getStory(...) }.
Risk
(a) WebView construction + HTML parse + first layout is heavy — visible flicker on fold/unfold of a foldable. (b) A JS-triggered native callback doing disk I/O on the UI thread is odd coupling and a potential ANR. Teardown itself is careful and correct (WebViewSafety.disposeAll + listener detach).
Fix
Make the reader's rerender only re-apply padding/column width (a layout pass), reserving full showReader rebuilds for chapter navigation. Move the storage.getStory read off the UI thread.
LowTheming · tokens ui/Tokens.kt:14-16,28-35 · many call sites

U8 · Spacing token migration is half-finished

Now

07Kotlin idioms & API design

The model layer and naming conventions. Some choices are deliberately pragmatic; others are latent footguns.

MediumIdioms · mutability domain/model/Models.kt:53-60,73-101,141-155,173-186,188-202

K1 · Mutable var data classes mutated both in-place and via copy

Now
Story/Chapter/DownloadJob/DisplayPreferences/TtsSession are data classes with all var fields. Code mutates them both ways: story.copy(...) in BackupMergePlanning/ArchiveSnapshotPlanning, and in-place story.dateAdded = ..., job.status = ....
Risk
data class equality/hashCode include those mutable fields, so a Story used as a map key or compared across mutations is unsafe. In-place mutation of objects held by StateFlow means a Story referenced by the UI can change underneath it without copy() — the very thing MutableStateFlow is supposed to make safe. The repo doc even admits models "may be updated in place by the download engine," which is why a downloadStateVersion counter was bolted on — a symptom of the mutability problem.
Fix
Pick one model: immutable val data classes + copy() everywhere. Given StateFlow usage, prefer val and always replace via copy.

Now

MediumDependency choice · Gson app/build.gradle:107 · data/storage/* · domain/model/Models.kt · source/PatreonStatsFetcher.kt

K4 · Gson instead of kotlinx.serialization

Now
Single GsonBuilder().setPrettyPrinting().create(); the whole domain model relies on Gson reflection. The team already hand-rolls wire-string compat (DownloadJobStatus.wire) to dodge Gson's enum-name coupling — a sign Gson is fighting them.
Risk
Gson is in maintenance mode. On Kotlin it (a) uses reflection that breaks under R8 unless you write keep rules (interacts with the minifyEnabled false finding), (b) doesn't understand Kotlin nullability/defaults reliably (a missing field can yield null in a non-null val), (c) generates no compile-time errors for schema drift, and (d) the Map<String,Any?> → toJson → fromJson<T> round-trips erase numeric types (usually to Double before coercing back). Current epoch-ms timestamps remain exactly representable, but the type erasure is needless and fragile.
Fix
Adopt kotlinx.serialization with @Serializable models + the compiler plugin. Compile-time-checked schemas, sealed-class polymorphism, R8-friendly. Migrate incrementally — the DownloadJobStatus.wire pattern maps cleanly to a custom serializer.

10Strengths to preserve

These are the things the codebase does right. Don't lose them while fixing the issues above.

StrengthTest design

S1 · The *Planning.kt functional-core pattern is exemplary

What
Side-effecting logic (file copies, network, storage mutations) is cleanly separated from pure decision logic. ArchiveSnapshotPlanning.buildArchiveSnapshot (a pure function) takes a copyChapter lambda as its only side-effect seam; the test injects an in-memory list and asserts the exact archive shape, including the fallback-when-copy-fails path. Same in DownloadScheduler.selectEligibleJobs, StoryActionGuards, CleanupEngine.
Why it matters
This is a disciplined "functional core, imperative shell." It's why 287 fast JVM tests exist with no Robolectric and no mocks, and it offsets the lack of DI — you don't need fakes if the logic under test takes pure inputs. Preserve and extend this convention.
StrengthConcurrency hygiene

S2 · Clean coroutine fundamentals

What
No GlobalScope, no executable production runBlocking, no Thread.sleep. StateFlows are exposed read-only via asStateFlow() with private backing MutableStateFlows. Scopes are tied to lifecycles (lifecycleScope, SupervisorJob() engines).
Why it matters
These are the most common coroutine footguns, and the codebase avoids all of them. The remaining concurrency issues (C1's synchronized-in-coroutines, C2's missing repeatOnLifecycle) are subtler and fixable incrementally.
StrengthArchitecture

S3 · The AppContainer service-locator is justified and well-scoped

What
A process-wide manual service locator attached to Application, with a documented motivation: a single shared container prevents duplicate engines racing on the same JSON files.
Why it matters
For an app with one Activity, a handful of foreground services, and no multi-module graph, a service locator is a reasonable trade-off: zero kapt/ksp cost, fast builds, no annotation-processing fragility, and a graph small enough to hold in your head. No change needed unless the app goes multi-module.
StrengthArchitecture

S4 · JSON-file persistence is a defensible choice for this data shape

What
All state is stored as JSON files with atomic writes + a versioned envelope. No Room/SQL.
Why it matters
For an archiver whose data is "a list of stories each with a list of chapters," small and append-mostly, JSON files are human-inspectable, trivially backup-able (the backup feature leans on exactly this), and avoid Room ceremony. The atomic-write + envelope discipline is good in intent. No urgent change — just fix the durability/recovery holes (P1–P4) and reconsider only if the library grows past a few hundred stories.
StrengthDocumentation

S5 · Exceptionally well-documented design decisions

What
Nearly every non-trivial class carries a KDoc explaining why a choice was made (single-owner invariant, atomic write rationale, the no-ViewModel trade-off, foldable handling). Comments reference constraints (R1-R9, M1-M3, parity gaps) consistently.
Why it matters
This is rare and valuable. The handful of stale-comment findings (P13) are worth fixing precisely because the docs are otherwise trustworthy — keep the bar high.
