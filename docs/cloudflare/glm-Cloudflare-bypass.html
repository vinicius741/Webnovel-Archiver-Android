Cloudflare Bypass Strategy for ScribbleHub on Android

GLM research — codebase audit, Mihon deep-dive, and a survey of bypass methods (mid-2026). Target: Webnovel-Archiver-Android. Compiled 2026-07-04.

Bottom line up front

The app's OkHttp client is blocked because it has the wrong TLS/HTTP/JS fingerprint, no cf_clearance cookie, and no cookie jar at all. The current "browser bypass" (a Chrome Custom Tab) cannot work — Custom Tabs never expose cookies to the host app. The proven fix, used by every Tachiyomi/Mihon fork, is to solve the challenge in an in-app WebView, read the cf_clearance cookie back out of android.webkit.CookieManager, and replay it on OkHttp with a matching User-Agent. Five candidate plans are detailed in § Plans; this document intentionally presents more than one approach.

1. The problem, exactly

Syncing and downloads from www.scribblehub.com fail because Cloudflare is challenging every request the app makes. The user reported that even the existing "open in browser" fallback no longer works. This section explains why before the doc proposes fixes.

Why the OkHttp client is blocked

Cloudflare does not trust the request because of a stack of fingerprints that the OkHttp client cannot match to a real browser:

  • TLS fingerprint (JA3/JA4). OkHttp on Android uses Conscrypt/BoringSSL, whose ClientHello is recognisably "Android OkHttp" — not Chrome. Cloudflare computes this before any header arrives, so the User-Agent string is irrelevant.
  • HTTP/2 fingerprint. Frame ordering and window sizes also betray the client library.
  • No JavaScript challenge solving. A real browser receives a JS payload from /cdn-cgi/challenge-platform/..., executes it, and posts back a token. OkHttp cannot run JS, so the challenge page is returned as HTML instead of the real content.
  • No cf_clearance cookie. Even if the user somehow solved the challenge, the app's OkHttp client has no cookie jar, so the clearance cookie would be discarded on the very next response.
  • Stale User-Agent. The hardcoded UA in NetworkRequests.kt is Chrome/114, which is multiple major versions behind current Chrome. Outdated UAs are scored higher-threat.

Why the "Open Browser" fallback does not work

The Custom Tab trap

feature/browser/BrowserScreen.kt launches a Chrome Custom Tab. Custom Tabs run inside Chrome's process and share Chrome's cookie jar — the host app cannot read those cookies. There is no API to do so on a non-rooted device. So even when the user solves the Cloudflare challenge in the Custom Tab and earns a cf_clearance, the next OkHttp retry still has no clearance cookie and is blocked again. The retry-on-resume logic in SourceAccessRetryCoordinator only re-runs the original action with no new credentials. This is the architectural gap that must be closed.

2. Current codebase state

Findings from a full read of the networking and source-provider code:

HTTP client — source/network/NetworkClient.kt

  • Single OkHttp client, process-wide via AppContainer. No Ktor, no Retrofit.
  • No cookieJar set — OkHttp uses its default in-memory jar that accepts no cookies. Every Set-Cookie response is silently dropped.
  • ScribbleHub gets a 1.5 s per-request throttle and up to 3 retries on 403/429 — pure politeness heuristics. A real Cloudflare challenge returns 403 and just exhausts these retries.
  • Hardcoded UA: Mozilla/5.0 (Linux; Android 10; K) ... Chrome/114.0.0.0 Mobile Safari/537.36 (stale).
val defaultClient: OkHttpClient =
    OkHttpClient.Builder()
        .connectTimeout(30, TimeUnit.SECONDS)
        .readTimeout(45, TimeUnit.SECONDS)
        .build()   // ← no cookieJar(...), no interceptors

Challenge detection — source/network/SourceAccessBlock.kt

Detection is solid and conservative. isChallengeResponse fires only on cf-mitigated: challenge or on a Cloudflare server header corroborated by structural challenge HTML (cf_chl, challenge-platform, <title>just a moment...</title>). It throws SourceAccessBlockedException, which the download pipeline marks SOURCE_BLOCKED / retryable=false. 

fun isChallengeResponse(headers: Headers, body: String): Boolean {
    if (headers["cf-mitigated"].equals("challenge", ignoreCase = true)) return true
    if (!headers["server"].equals("cloudflare", ignoreCase = true)) return false
    return isChallengeHtml(body)
}

ScribbleHub provider — source/ScribbleHubProvider.kt

  • Jsoup-based parsing of metadata + chapters; no headless browser.
  • TOC pagination hits wp-admin/admin-ajax.php with a hardcoded Cookie: toc_show=50 header — the only cookie usage anywhere in the codebase.
  • Wraps TOC fetches in SourceAccessBlockedException handling so a mid-pagination block degrades gracefully.

Browser surface — feature/browser/

  • BrowserScreen.kt opens a Custom Tab. Receives only a URL back via the Import action — no cookies.
  • SourceAccessActions.kt shows the "blocked" dialog and arms a one-shot retry on onResume().
  • In-app WebViews (ui/WebViewSafety.kt) exist only for the Reader and the import surface; none has a WebViewClient or any CookieManager.getInstance() call.
Note on android/AGENTS.md

The android-level instructions explicitly warn: "do not move third-party OAuth back into a WebView." A Cloudflare-solving WebView is a narrower concern than OAuth, but anyone implementing Plan A should reconcile this with the existing rule. The CookieManager-based cookie jar is the clean way to keep one shared store.

3. How Cloudflare blocks in 2026

Understanding the layers matters because every bypass method attacks a specific layer and most fail because they miss one. Ranked from cheapest to hardest for the attacker to fake:

  1. IP reputation / ASN. Datacenter IPs (AWS, GCP, Hetzner, DO) and known VPN egresses get a high threat score before any bytes are exchanged. Residential and mobile-carrier ASNs score lowest. Your phone has the highest-trust category by default.
  2. TLS fingerprint (JA3 → JA4). Hash of the ClientHello (cipher suites, extensions, curves, ordering). Computed before any HTTP header. An OkHttp ClientHello does not match a Chrome ClientHello.
  3. HTTP/2 fingerprint. Frame ordering and SETTINGS window sizes betray the client library.
  4. JavaScript challenge. A JS payload probes navigator, canvas, WebGL, timing, etc. The old arithmetic challenge is deprecated; current challenges are heavier and often culminate in Turnstile.
  5. Turnstile. Cloudflare's CAPTCHA replacement — invisible/managed, non-interactive, or interactive. Produces a token bound to the solving session.
  6. Session binding. The cf_clearance cookie is pinned to (UA + IP + fingerprint + a time window). Reuse from a mismatched client/IP and Cloudflare re-challenges.
Key consequence

A method that fixes only one layer (e.g. curl-cffi fixes TLS+HTTP/2 but not JS) will pass on low-protected sites and fail on sites with JS challenges or Turnstile. ScribbleHub runs a JS/Turnstile-grade challenge, so a real browser engine is effectively required — which is exactly why the WebView approach wins.

4. How Mihon does it (deep dive)

mihonapp/mihon (the maintained successor to Tachiyomi) is the reference implementation of Cloudflare handling for Android novel/manga readers. The design is WebView-as-an-oracle layered on top of the normal OkHttp stack.

The four moving parts

NetworkHelpercore/common/.../network/NetworkHelper.ktBuilds the shared OkHttp client, wires interceptors, owns the cookie jar.
CloudflareInterceptorcore/common/.../network/interceptor/CloudflareInterceptor.ktDetects a CF challenge, spins up a background WebView to solve it, retries.
AndroidCookieJarcore/common/.../network/AndroidCookieJar.ktOkHttp CookieJar backed by android.webkit.CookieManager.

The key insight: one shared cookie store

AndroidCookieJar implements OkHttp's CookieJar directly against android.webkit.CookieManager.getInstance(). Because WebViews also read/write that same CookieManager, the moment the WebView earns a cf_clearance cookie, it is instantly visible to OkHttp — no explicit copy step required. Persistence across app restarts is free, because CookieManager writes to disk in the WebView data directory. 

class AndroidCookieJar : CookieJar {
    private val manager = CookieManager.getInstance()  // android.webkit.CookieManager

    override fun saveFromResponse(url: HttpUrl, cookies: List<Cookie>) {
        cookies.forEach { manager.setCookie(url.toString(), it.toString()) }
    }

    override fun loadForRequest(url: HttpUrl): List<Cookie> {
        val cookies = manager.getCookie(url.toString()) ?: return emptyList()
        return cookies.split(";").mapNotNull { Cookie.parse(url, it) }
    }
}

Detection (shouldIntercept)

private val ERROR_CODES = listOf(403, 503)
private val SERVER_CHECK = arrayOf("cloudflare-nginx", "cloudflare")

override fun shouldIntercept(response: Response): Boolean {
    return if (response.code in ERROR_CODES &&
        response.header("Server") in SERVER_CHECK) {
        val document = Jsoup.parse(response.peekBody(Long.MAX_VALUE).string(), ...)
        // only fire on a real challenge, NOT on a geo block
        document.getElementById("challenge-error-title") != null ||
            document.getElementById("challenge-error-text") != null
    } else false
}

Compare to the Webnovel-Archiver SourceAccessBlockDetector — already in the same shape, but it throws instead of triggering a solve. That is exactly the seam a bypass plugs into.

The background solve (resolveWithWebView)

This is the "helper WebView" you asked about. Mihon creates a transient, off-screen WebView on the main thread, loads the blocked URL, lets the system WebView solve the JS challenge, waits for the cf_clearance cookie to appear, then destroys the WebView. It is never attached to any view hierarchy and the user never sees it. 

private fun resolveWithWebView(originalRequest: Request, oldCookie: Cookie?) {
    val latch = CountDownLatch(1)
    var webview: WebView? = null
    var cloudflareBypassed = false

    executor.execute {  // main thread — WebViews must be touched on the main thread
        webview = createWebView(originalRequest)
        webview!!.webViewClient = object : WebViewClient() {
            override fun onPageFinished(view: WebView, url: String) {
                val newCookie = cookieManager.get(origRequestUrl.toHttpUrl())
                    .firstOrNull { it.name == "cf_clearance" }
                // ★ success criterion: a cf_clearance that differs from the one before the solve
                if (newCookie != null && newCookie != oldCookie) {
                    cloudflareBypassed = true
                    latch.countDown()
                }
            }
        }
        webview!!.loadUrl(origRequestUrl, headers)
    }

    latch.awaitFor30Seconds()   // hard 30-second cap
    executor.execute { webview?.run { stopLoading(); destroy() } }
    if (!cloudflareBypassed) throw CloudflareBypassException()
}

Success criterion (the part you specifically asked about): Mihon does not poll, does not detect a redirect, and does not check for the absence of the challenge page. It checks, in onPageFinished, whether a cf_clearance cookie now exists that is different from the one captured before the solve. The interceptor deletes the old cookie first, so any cf_clearance present after the page load must be freshly granted.

User-Agent matching — the one rule you cannot break

Cloudflare binds cf_clearance to (at minimum) the client IP and the User-Agent. Mihon enforces consistency by construction:

  • WebViewInterceptor.createWebView sets settings.userAgentString to the request's UA header.
  • HttpSource.headersBuilder() adds the shared default UA to every request.
  • The default UA is a normalised Chrome-on-Android string (device model masked as Android 10; K, the Version/ token stripped) so it does not leak device fingerprint.
// WebViewUtil.getInferredUserAgent normalises the WebView UA:
//   "...; Android 13; Pixel 7 Build/...; wv) ... Version/4.0 Chrome/116..."
// becomes
//   "...; Android 10; K) ... Chrome/116..."
.defaultUserAgentString()
    .replace("; Android .*?\\)".toRegex(), "; Android 10; K)")
    .replace("Version/.* Chrome/".toRegex(), "Chrome/")

5. All methods surveyed

For each method: how it works, whether it still works against current CF, and fit for this app.

1. In-app WebView cookie capture recommended

The Mihon/Tachiyomi pattern (§4). Open the protected URL in an android.webkit.WebView; the system Chromium solves the challenge for real; read cf_clearance + UA out of CookieManager; replay on OkHttp with a matching UA. The WebView is a genuine browser, so TLS/JA4, HTTP/2, JS, and fingerprint checks all pass natively.

Effectiveness: high · Android effort: low–medium · Maintenance: low · Server: no · ToS: lowest.

2. Chrome Custom Tabs / system browser not viable

Launching the URL in the system Chrome would solve the challenge, but there is no API to read cookies from Custom Tabs or system Chrome on a non-rooted device. The cookie jar belongs to Chrome's process and lives in Chrome's sandboxed storage. This is the design property that makes Custom Tabs "more secure" — and exactly why our current fallback fails. Trusted Web Activity (TWA) + Digital Asset Links does not change this.

Effectiveness: N/A (no cookie access) · This is what the app does today.

3. FlareSolverr officially broken

A Docker'd proxy that runs a headless Chromium (historically undetected-chromedriver) and exposes a JSON API: {"cmd":"request.get","url":"..."} → returns solution.cookies + solution.userAgent + HTML. TRaSH Guides states bluntly: "FlareSolverr is currently non-functional. It is also being monitored by the Cloudflare team, so it is unlikely to ever be fixed." The pattern (self-hosted solver) remains valid — see method 7 — but FlareSolverr itself is dead against current CF.

Effectiveness: low · Server: required · Maintenance: high. Also has an IP-binding problem: the cf_clearance is minted from the server's IP, so the phone would have to route its HTTP through the same server to reuse it.

4. undetected-chromedriver / selenium-driverless / Playwright-stealth / Puppeteer-stealth mostly obsolete

The 2020–2023 generation of "stealth" automation. Modern CF catches them on CDP/WebDriver signals; they leak enough tells that the arms race has moved on. None run on Android — they are server-side only. The 2026 benchmark shows them well behind the newer tools in §7.

Effectiveness: low–medium and declining · Server: required · Maintenance: high.

5. cloudscraper / cffi / cf_clearance libraries (pure HTTP) dead

cloudscraper parsed Cloudflare's old arithmetic JS challenge (v1). The v1 challenge is deprecated and cloudscraper's solver is flagged deprecated on PyPI. There is no maintained JVM port, and against a Turnstile-grade site like ScribbleHub it will not get in. The "cf_clearance libraries" that do work are misnamed — they wrap a real browser, so they are really method 1 or 7 in a trenchcoat.

Effectiveness: none against ScribbleHub-class CF. Do not spend time here.

6. TLS fingerprint spoofing (uTLS / CycleTLS / curl-cffi / tls-client) necessary, not sufficient

These spoof the TLS ClientHello to look like Chrome/Firefox byte-for-byte, fixing layer 2. They defeat the TLS gate on low-protection sites but do not solve the JS challenge — you still get the challenge page back as HTML. And there is no production-grade Kotlin/JVM uTLS equivalent: OkHttp on Android uses Conscrypt, whose ClientHello is recognisably "Android OkHttp." On-device, let the WebView handle TLS for you (the cookie is what matters). Server-side, curl-cffi is the most popular and best-maintained option.

Effectiveness: medium alone, essential as a component · Kotlin: high effort · Server: only for pure-HTTP spoofing.

7. Modern stealth browsers: nodriver / Patchright / Camoufox / DrissionPage / puppeteer-real-browser best-in-class (server-side)

The 2024–2026 successors. nodriver drives Chrome over raw CDP with no Selenium/Playwright/WebDriver footprint at all, and passed every Cloudflare gate cleanly in the benchmark. Camoufox is a custom Firefox with C++-level fingerprint injection. Patchright is a patched Playwright that fixes CDP leaks. All are server-side Python/Node tools — none run on Android. They are the right answer if you ever need to replace FlareSolverr with a self-hosted solver: a tiny FastAPI wrapper around nodriver that returns {cf_clearance, userAgent, html}.

Effectiveness: high (nodriver/Camoufox/Patchright) · Server: required · Maintenance: medium.

8. Residential / mobile proxies mostly free for this app

IP reputation matters a lot — datacenter ASNs are penalised. But the app's users are already on residential/mobile-carrier IPs, the highest-trust category, and the WebView approach (§1) solves from that same clean IP. Paid proxies only enter the picture for a server-side deployment that needs clean residential egress. Note: cf_clearance is bound to the IP that solved the challenge — pin the IP for the lifetime of the cookie; rotate only on re-solve.

Effectiveness: enabler, not a solution · Cost: high if paid · Usually unnecessary on-device.

9. Cloudflare Turnstile solving services token-replay limits

A real browser (incl. Android WebView) auto-passes invisible/managed Turnstile. For interactive Turnstile you can call a CAPTCHA-solving service (2Captcha, CapSolver, AntiCaptcha): POST the sitekey + page URL, get a cf-turnstile-response token back (~$1–3 / 1000 solves). The catch: the token is bound to the solving session/IP/TLS, so pure token replay from a mismatched client often fails verification. For this app you almost certainly do not need to handle Turnstile explicitly — WebView renders it and it auto-solves on a clean mobile IP.

Effectiveness: medium (token-replay limits) · Android: low via WebView · Maintenance: medium if used.

10. Official API / RSS / alternate endpoints also CF-protected

ScribbleHub exposes RSS at /rssfeed.php?type=.... The ScribbleHub forum itself has a thread titled "RSS feeds getting caught in Cloudflare again." A live probe (2026-07-04) returned HTTP 403 for rssfeed.php, /rssfeed.php?type=main, and /feed/ — i.e. RSS is behind the same wall. There is no public REST API and no api.scribblehub.com; the site is WordPress-based.

Useful optimisation, not a bypass: once you have solved CF via WebView and hold a valid cf_clearance, polling rssfeed.php for chapter updates is far cheaper than scraping HTML — lower bandwidth, lower re-challenge risk. Use it as a payload optimisation after the bypass, not a way around it.

7. Recommended plans (multiple)

As requested, this is not a single recommendation. Each plan is a complete, buildable approach with its integration points in this codebase. Plan A is the recommended primary; the others are alternatives if Plan A proves insufficient or you want a different posture.

Plan A — In-app WebView cookie capture (the Mihon pattern)

Recommended Lowest risk

On-device · No server · Lowest ToS exposure · Battle-tested by every Tachiyomi/Mihon fork.

What it does. Replaces the Custom Tab fallback with an in-app WebView that solves Cloudflare, and adds a persistent OkHttp cookie jar so the earned cf_clearance is reused on every subsequent request.

New / changed files

  • source/network/AndroidCookieJar.kt (new) — OkHttp CookieJar backed by CookieManager.getInstance(). ≈ 30 lines, modelled on Mihon's.
  • source/network/CloudflareBypassInterceptor.kt (new) — OkHttp Interceptor. On a detected challenge, launches a hidden WebView, waits up to 30 s for a new cf_clearance, then retries chain.proceed(request).
  • feature/browser/CloudflareSolveActivity.kt (new, optional) — interactive fallback WebView the user can solve by hand if the background solve times out.
  • source/network/NetworkClient.kt — attach .cookieJar(AppContainer.cookieJar) and .addInterceptor(CloudflareBypassInterceptor(...)) on defaultClient.
  • source/network/NetworkRequests.kt — bump the UA to a current Chrome major (e.g. Chrome/141), and make it the same string the WebView uses.
  • feature/browser/SourceAccessActions.kt — repoint the "Open Browser" button at the in-app CloudflareSolveActivity instead of a Custom Tab.

Build sequence

  1. Add the cookie jar on AppContainer; wire it into NetworkClient.defaultClient. Verify existing tests still pass (no behaviour change yet for non-CF sites).
  2. Add CloudflareBypassInterceptor with the background-WebView solver. Key invariants: WebView UA = OkHttp UA; delete old cf_clearance before solving; success = a new cookie appears in onPageFinished; 30 s hard timeout.
  3. Make SourceAccessBlockDetector throw → interceptor, not the UI. Today the detector throws on first sight; with the interceptor upstream, the throw only fires if the solve also fails.
  4. Add the interactive fallback activity and repoint the dialog button.
  5. Refresh NetworkRequests.USER_AGENT; add a unit test that asserts OkHttp UA equals the WebView UA used by the solver.
Beats current CF No server Low maintenance Needs ≥ WebView 118 Cookie tied to UA+IP

Plan B — Hybrid: WebView solve + RSS polling

Plan A + payload optimisation

On-device · No server · Lower bandwidth and re-challenge risk for follow-up syncs.

What it does. Plan A exactly, plus: once a cf_clearance is held, poll https://www.scribblehub.com/rssfeed.php?... with the cookie for incremental chapter updates instead of scraping every novel's HTML TOC. RSS is far lighter and keeps the clearance cookie "warm" with lower request volume.

Why this is worth the extra work

  • The full sync currently walks many TOC pages per novel; each fetch is a chance for Cloudflare to re-challenge. RSS collapses a per-novel crawl into a single feed read.
  • Lower bandwidth, faster syncs, friendlier to the source.

Caveat

RSS is also behind Cloudflare (verified 403 today), so it is not a way to avoid solving CF — it is an optimisation layered on top of Plan A. If RSS format changes or stops being useful, fall back to HTML scraping under the same cookie.

Beats current CF No server Lower bandwidth RSS is fragile

Plan C — Self-hosted solver microservice (nodriver)

Server-side

Requires the user to run a small service on a home PC / NAS · Modern FlareSolverr replacement.

What it does. A tiny FastAPI wrapper around nodriver running on the user's own machine. The Android app POSTs the blocked URL; the service solves CF in a real headless Chrome and returns {cf_clearance, userAgent, html}; the app replays the cookie on OkHttp.

Why you might pick this

  • Works even on devices where the system WebView is outdated or unavailable.
  • Solves once and shares the cookie across devices on the same home network.
  • sidesteps interactive challenges the on-device WebView cannot solve unattended.

Why it is not the primary plan

IP-binding trap

cf_clearance is bound to the IP that solved it. If the solver runs on the home server but the phone is on cellular, the cookie is rejected. For this to work the phone must either be on the same network as the solver or route its HTTP through the solver (making the solver a forwarding proxy). That doubles the build complexity.

  • Requires the user to install and run software off-device — a major UX regression for a personal reader app.
  • Extra maintenance: track Chrome versions, keep the service running.

When to reach for it

Only if Plan A proves unable to solve a specific ScribbleHub challenge on a meaningful share of devices.

Beats current CF Server required IP-binding UX cost

Plan D — Minimal fix: persistent cookie jar + Custom-Tab-as-solver prompt

Smallest change

On-device · No server · Works only because the user solves manually · Fragile.

What it does. The smallest possible delta from today. Add a persistent AndroidCookieJar to OkHttp (the same one as Plan A) but do not build the in-app solver. Instead, when a block is detected, prompt the user to solve in the system browser, then read the resulting cookie out of CookieManager.

This does not actually work

Chrome Custom Tabs / system Chrome do not write their cookies into the app's CookieManager — they write into Chrome's own store. So after the user solves in the browser, CookieManager.getInstance().getCookie("www.scribblehub.com") still returns null. This plan is listed only to document why "just add a cookie jar" is not enough on its own: you need the WebView (Plan A) for the cookie to land in the right jar.

Does not bypass CF Tiny change Fragile / manual

Plan E — Turnstile CAPTCHA-solving service integration

Paid third party

On-device · Recurring cost · Only relevant if ScribbleHub forces interactive Turnstile.

What it does. Integrate a CAPTCHA-solving service (2Captcha, CapSolver, AntiCaptcha). When an interactive Turnstile is detected, POST the sitekey + page URL; the service returns a cf-turnstile-response token; submit it. ~$1–3 per 1000 solves.

Why this is last-resort only

  • Cloudflare's managed/invisible Turnstile auto-passes in a real WebView, so Plan A handles it for free. This plan is only needed if ScribbleHub escalates to forced interactive Turnstile.
  • Tokens are bound to the solving session/IP/TLS — pure token replay from a mismatched client often fails verification. So this typically still needs a WebView context to submit, which puts you back in Plan A's territory anyway.
  • Recurring cost, additional ToS surface, dependency on a third party.
Conditional need Recurring cost Token-replay limits

Recommendation summary

Primary: Plan A

Ship the in-app WebView → CookieManager.getCookie() → OkHttp-with-matching-UA flow. It is the only approach that is on-device, low-maintenance, legitimately defensible, and proven by the entire Tachiyomi/Mihon ecosystem. The cookie-jar + interceptor architecture maps cleanly onto the existing SourceAccessBlockDetector seam and the AppContainer singleton.

Recommended enhancement: Plan B (after A is stable)

Add RSS polling for incremental chapter updates once the CF cookie is held. Lighter, faster, and reduces re-challenge risk.

Fallback only: Plans C / E

Reserve the server-side nodriver service (Plan C) and the paid Turnstile service (Plan E) for the case where Plan A demonstrably cannot solve a specific challenge on a real share of user devices. Plan D is documented only to explain why the naive "just add a cookie jar" idea fails.

Do not spend time on

cloudscraper (dead v1 solver), FlareSolverr (officially broken), Chrome Custom Tabs cookie extraction (no API), reimplementing uTLS in Kotlin (no library, enormous effort), or buying residential proxies (users already have ideal mobile IPs).

8. Legal & ToS notes

  • Cloudflare ToS. Restricts circumventing their security services in commercial/abusive contexts. The legal weight against an individual scraping their own user account for personal use is negligible; no clear US case law criminalises solving a JS challenge to access data the user is authorised to see.
  • ScribbleHub ToS. Check the site ToS for scraping clauses. Scraping a user's own followed-list for personal, non-redistributive use (a reader app) is low-risk and squarely within how Tachiyomi/Mihon extensions have operated for years for similar sites.
  • CFAA (US) / Computer Misuse Act (UK). The dominant risk factor is accessing data you are not authorised to. The user is logged into their own account and fetching their own readlist — authorisation is present. Avoid redistributing scraped text content publicly.
  • Cleanest posture. The WebView approach (Plan A) is a real browser, driven by the real user on their real device, accessing their own account. It is the least "circumvention-y" option available — strongly preferable on both technical and legal grounds to CAPTCHA-solving services (Plan E) or proxies.
