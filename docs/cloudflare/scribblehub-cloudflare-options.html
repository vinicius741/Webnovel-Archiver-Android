Scribble Hub Cloudflare Access Options

Investigation and implementation recommendations for making Scribble Hub sync and downloads work when Cloudflare presents a browser challenge. Written for the native Kotlin Android app in this repository.

Prepared: 2026-07-04

Compliance boundary. This document focuses on user-mediated access: detect Cloudflare challenges, let the user complete the challenge in a real Android browser context, then reuse the resulting first-party session cookie inside the app. It does not recommend CAPTCHA solving services, proxy rotation, headless browser automation, challenge reverse engineering, fingerprint spoofing libraries, or other traffic-evasion techniques. Those are brittle, likely to violate site expectations, and can create user and project risk.

Executive Summary

The current workaround cannot fix downloads because the app opens Scribble Hub in a Custom Tab or external browser, but the downloader uses a separate OkHttp client with no shared cookie jar. Even if the user passes Cloudflare in Chrome, the OkHttp requests made by NetworkClient do not receive that browser's cf_clearance cookie.

Mihon's successful pattern is different: OkHttp and WebView share Android's android.webkit.CookieManager. When a challenge is detected, Mihon opens a WebView using the same request headers, waits for a fresh cf_clearance cookie, and retries the OkHttp request. Mihon also exposes a manual WebView screen and WebView data/cookie reset tools.

Recommended plan: implement a shared WebView/OkHttp cookie store first, add an in-app "Source access" WebView flow for Scribble Hub, then add a carefully bounded Mihon-style automatic retry only after the user initiates access. Add source-specific throttling and diagnostics so the app avoids triggering challenges in the first place.

Current App Findings

Area Current behavior Why it fails
Challenge detection SourceAccessBlockDetector checks cf-mitigated: challenge, or server: cloudflare plus strong challenge HTML markers. This is directionally correct and matches Cloudflare's documented cf-mitigated signal. Detection is not the main problem.
Browser workaround showSourceAccessBlockedDialog() sends the user to showBrowser(), which launches an Android Custom Tab and falls back to ACTION_VIEW. Custom Tabs share the user's browser state, not this app's OkHttp cookie store. The downloaded chapters still come from OkHttp and remain unauthenticated from Cloudflare's perspective.
OkHttp client NetworkClient.defaultClient sets timeouts only. It has no CookieJar. OkHttp will not persist Set-Cookie values or send stored WebView cookies unless a cookie jar is provided.
Scribble Hub source Metadata and chapter fetches are direct HTTP requests. The table-of-contents AJAX request adds a toc_show=50 cookie manually, but there is no general site cookie store. The source can parse real pages, but it cannot carry a Cloudflare browser clearance session.

What Cloudflare Expects

Cloudflare Challenges are designed to verify that a visitor is a real human and not automated traffic. Official docs say challenge responses include cf-mitigated: challenge, and the response content type is text/html regardless of the requested resource type.

When a visitor solves a challenge, Cloudflare sets a cf_clearance cookie. That cookie is tied to the visitor and device and is evaluated on later requests until it expires. Cloudflare's default challenge passage is 30 minutes, and its docs recommend 15 to 45 minutes. This means the app should treat clearance as temporary, per-device, and non-portable.

Cloudflare officially supports current major desktop and mobile browsers, with limited support for custom browser engines, WebViews, and embedded browsers. It explicitly does not support command-line tools, headless browsers, or automation frameworks for challenge solving. This is why direct OkHttp cannot "solve" a challenge by itself.

How Mihon Handles It

Core architecture

  • Mihon's NetworkHelper builds one OkHttp client with an AndroidCookieJar and a CloudflareInterceptor.
  • AndroidCookieJar implements OkHttp CookieJar by delegating to android.webkit.CookieManager.getInstance(). WebView and OkHttp therefore read and write the same Android WebView cookie store.
  • Its Cloudflare interceptor detects a 403/503 response from Cloudflare, removes the old cf_clearance, creates a WebView on the main thread, loads the original URL with safe request headers, waits up to 30 seconds for a new cf_clearance, destroys the WebView, and retries the original OkHttp request.

Supporting pieces

  • WebViews are configured with JavaScript, DOM storage, multiple-window support, zoom controls, and third-party cookie acceptance.
  • Mihon derives a realistic Android Chrome-style default user agent from the installed WebView and lets users override/reset the default user agent in Advanced settings.
  • Mihon exposes "Open in WebView", "Clear cookies", and "Clear WebView data" flows. Its manual WebView screen also detects Cloudflare challenge markers and links to troubleshooting help.
  • Mihon has patched WebView's X-Requested-With package-name behavior in recent releases because some sources block identifiable embedded-browser requests. This is a compatibility hack, not the first thing this project should copy.

Approach Options

Option How it works Pros Cons and risk Recommendation
A. Shared WebView/OkHttp cookie jar
Mihon-likeHigh value 		Add an app-level AndroidCookieJar backed by CookieManager, pass it to OkHttpClient.Builder.cookieJar(), and make all source fetches use that client. Create a Scribble Hub source-access WebView that the user can open when challenged. Fixes the current architecture gap. Lets user-solved first-party cookies apply to sync, chapter downloads, cover downloads, and AJAX table-of-contents requests. WebView challenge solve is not guaranteed on all devices. Cookies expire. Needs privacy-conscious settings and clear cookie controls. Do first. This is the foundation for every legitimate path.
B. Explicit in-app Source Access screen
User mediatedTransparent 		Replace the Custom Tab retry path with an in-app WebView screen for Scribble Hub. The screen shows the real page, waits until the page is no longer a challenge and/or a cf_clearance cookie is visible, then returns to retry the blocked action. Keeps the human challenge in front of the user. Easier to debug than a hidden WebView. Gives the app access to the same cookie store used by OkHttp. More UI work than the current dialog. Some challenges may have limited WebView support. Do with Option A. Prefer this before hidden automatic solving.
C. Mihon-style interceptor retry
ConvenientNeeds guardrails 		When OkHttp receives a Cloudflare challenge, open a temporary WebView on the main thread, load the original URL using safe headers and the request user agent, wait for a new cf_clearance, then retry once. Reduces friction after a user has already allowed source access. Handles background sync/download challenges without forcing every request through the visible browser. Blocking an interceptor for up to 30 seconds is fragile. Hidden WebViews can look more automated. Must avoid loops and must not run indefinitely from background services. Second phase. Use only after Option A/B exist. Limit to one attempt per host, short timeout, foreground/user-initiated actions first, and clear failure UI.
D. WebView content fallback
FallbackHeavy 		For Scribble Hub only, load story/chapter pages in an in-app WebView after user clearance, then extract the final DOM HTML with evaluateJavascript and feed existing Jsoup parsers. Works when the site requires JavaScript before content is available. Can reuse current parsers after DOM extraction. Slow, memory-heavy, main-thread sensitive, hard to run safely for bulk downloads. Requires careful lifecycle handling and cancellation. Fallback only. Useful for "download this one chapter now", not a default bulk sync engine.
E. Avoid triggering challenges
PreventiveLow risk 		Add per-source policies: lower concurrency for Scribble Hub, larger request gaps, honor Retry-After, cache metadata/table-of-contents pages, stagger queued downloads, and stop hammering after the first challenge. Reduces challenge frequency and protects the user's IP reputation. Improves reliability even without Cloudflare. Does not solve an active challenge by itself. Slower downloads. Do in parallel. This should be part of the production-quality fix.
F. Custom Tab only
Current path 		Open Scribble Hub in Chrome Custom Tabs or an external browser and ask the user to return. Uses a fully supported browser. Good for import URL browsing. Does not share cookies with app-owned OkHttp. Cannot fix sync/download unless the app switches networking to the browser context, which Android does not expose for arbitrary HTTP clients. Do not rely on this for downloads. Keep it for "open in browser".
G. Anti-detect/proxy/CAPTCHA-service bypass
Do not implement 		Rotate residential proxies, emulate browser fingerprints, use headless automation, outsource challenges, or reverse engineer challenge scripts. None appropriate for this project. Likely violates site expectations, breaks often, can harm users, and conflicts with Cloudflare's stated unsupported environments. Reject. Do not add these paths.

Recommended Implementation Plan

  1. Create a process-wide source network module. Move NetworkClient.defaultClient construction into AppContainer and provide one shared OkHttp client for pages, AJAX, and binary downloads.
  2. Add an Android WebView cookie jar. Implement a small CookieJar that reads and writes CookieManager.getInstance(), following Mihon's shape. Make sure CookieManager.setAcceptCookie(true) is enabled, use setCookie() for response cookies, parse getCookie() for request cookies, and call flush() after important source-access completions.
  3. Replace the blocked-source dialog action. For Scribble Hub challenges, open an in-app "Source access" WebView instead of a Custom Tab. Enable JavaScript and DOM storage, keep file/content access disabled, and use a user agent consistent with the OkHttp requests.
  4. Detect success explicitly. Treat access as passed when CookieManager.getCookie() for https://www.scribblehub.com/ includes cf_clearance= and the loaded page no longer looks like a Cloudflare challenge. Then retry the original sync/download once.
  5. Add settings and diagnostics. Add "Clear Scribble Hub cookies", "Clear WebView data", current WebView version display, and better error text when WebView is missing or outdated. Include the blocked URL and whether the response had cf-mitigated.
  6. Add source throttling. Replace hard-coded Scribble Hub checks inside NetworkClient with a source network policy. Start conservative: one request at a time, 3-5 seconds between Scribble Hub requests, honor Retry-After, and pause a queue immediately after a challenge.
  7. Consider hidden WebView retry only after manual access works. If we copy the Mihon-style interceptor, constrain it: one attempt, 20-30 second timeout, no infinite retry, foreground notification for background downloads, and a fallback to the visible Source Access screen.
  8. Keep Custom Tabs as a separate browser feature. Use Custom Tabs for importing URLs and "open externally", not as the Cloudflare fix for app-managed downloads.

Design Notes for This Codebase

  • Keep deterministic decisions in pure planning helpers. For example, challenge detection, retry eligibility, and source-policy timing can have focused unit tests.
  • Put WebView lifecycle work under feature/browser/ or a small source/access/ package, but keep source parsing inside source/ScribbleHubProvider.kt.
  • Do not reuse the reader WebView settings blindly. The source-access WebView loads untrusted remote pages and needs stricter file/content access boundaries than the sanitized reader.
  • Use the emulator debug variant for QA. Cloudflare may behave differently on an emulator than on a real device, so avoid claiming physical-device reliability unless explicitly tested with current permission.

Acceptance Criteria

  • When a Scribble Hub request returns cf-mitigated: challenge, the app opens a visible in-app WebView source-access flow.
  • After the user passes the challenge, CookieManager.getCookie("https://www.scribblehub.com/") exposes a cf_clearance cookie and the next OkHttp request sends it automatically.
  • Retrying story sync succeeds without reinstalling the app or switching to the release package.
  • Download queue jobs pause on challenge instead of repeatedly hammering Scribble Hub.
  • The user can clear Scribble Hub cookies/WebView data from settings or the source-access screen.
  • Unit tests cover challenge detection, retry eligibility, cookie-header parsing, and source throttling decisions. Instrumented/emulator QA covers the visible WebView flow.
