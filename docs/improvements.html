Where the Architecture Can Go Next

This is a companion to the architecture reference. The codebase is already well-disciplined — a clean planning/engine split, durable storage, and a single-owner model. The proposals below are evolutions, not rewrites: each one targets a concrete, verified gap, with an effort/impact rating and a sketch of the target state. Tap any proposal to expand the detail.

00How to Read This

Every proposal was checked against the actual source, not generated from a checklist. Here's the convention:

Rating scale

Each proposal carries two meters:

Impact High Effort Small
  • Impact — how much it improves reliability, speed, or maintainability
  • Effort — S (hours) · M (days) · L (weeks) · XL (careful, multi-PR)

Each proposal contains

  • The gap — what's wrong today, with a file/line reference
  • The fix — the target design, with a before/after sketch
  • Trade-offs — what it costs and what to watch for
  • First step — the smallest meaningful change to start

These are proposals, not a mandate. The codebase explicitly chose several "unusual" patterns (no DI framework, no WorkManager, no XML) for good reasons documented in code comments. Each proposal below respects those choices — none of them reintroduces what was deliberately left out.

01What's Already Strong

Before changing anything, it's worth naming what should be preserved. These are the load-bearing decisions the rest of the proposals build on top of, not around.

Planning / Engine split

Pure, Android-free *Planning objects make deterministic logic trivially testable. Keep this. Several proposals extend it.

Durability primitives

DurableJson + AtomicFileWrites (temp→fsync→rename) are correct. Proposals add tests, not replace them.

Single-owner storage

One AppStorage + monitor serialization prevents cross-component races. The model is sound; P1 fixes how it's awaited.

No XML UI

The programmatic View DSL keeps screens as pure functions of state. Worth keeping and lean into (P4, P9).

Source provider seam

The SourceProvider/SourceRegistry dispatch table is clean. P11 widens it, doesn't restructure it.

Transactional restore

The staged-swap backup restore is genuinely well-engineered (zip-slip protection, size caps, snapshot). Little to change.

Reliability

P1 Replace `synchronized(storage)` inside `suspend` functions with a coroutine `Mutex` verified gap Impact Effort ▶

The gap

AppRepository declares its transactional mutations as suspend but guards them with a blocking JVM monitor (synchronized(storage)) — 7 occurrences in AppRepository.kt plus 3 in DownloadEngine.kt. A blocking monitor inside a suspend function is a coroutine anti-pattern: it pins the dispatcher thread for the duration of disk I/O, and if such a function is ever awaited from Dispatchers.Main it can stall the UI. The download code comments explicitly call out avoiding runBlocking for ANR reasons — but the repository uses the same blocking primitive one layer up.

Before / after

● Now — AppRepository.kt
suspend fun updateStory(
    storyId: String,
    block: (Story?) -> Story?,
) {
    synchronized(storage) {
        val current = storage.getStory(storyId)
        val updated = block(current) ?: return@synchronized
        storage.addOrUpdateStory(updated)
        _libraryFlow.value = storage.getLibrary()
    }
}
✦ Proposed
private val mutex = Mutex()

suspend fun updateStory(
    storyId: String,
    block: suspend (Story?) -> Story?,
) {
    mutex.withLock {
        val current = storage.getStory(storyId)
        val updated = block(current) ?: return@withLock
        storage.addOrUpdateStory(updated)
        _libraryFlow.value = storage.getLibrary()
    }
}

Trade-offs

  • Pro: No thread is pinned; the calling dispatcher is freed while waiting. The suspend block can now itself call suspend functions (useful for P2).
  • Pro: Eliminates a latent ANR path if any UI flow ever calls these from Main.
  • Con / care: The download engine intentionally uses mutateQueueInPlace (non-suspending @Synchronized) to stay off coroutines on the hot path — that blocking is acceptable because it runs on Dispatchers.IO and the critical section is tiny. P1 only changes the suspend surface in AppRepository, where the contract already promises suspension.
  • Con: A single shared Mutex serializes all repo transactions — same as today, so no behavior change, but document it.

First step

Introduce private val storageMutex = Mutex() in AppRepository, swap the 7 synchronized(storage) blocks in the suspend methods to storageMutex.withLock, and add a unit test asserting two concurrent updateStory calls don't interleave. Leave DownloadEngine's non-suspending queue path untouched.

P2 Make the engines unit-testable — they currently have zero test coverage verified gap Impact Effort ▶

The gap

The test suite is lopsided: 52 test files for 101 source files, but every test targets a *Planning object or a pure helper. There are no tests for DownloadEngine, StorySyncEngine, EpubEngine, TtsEngine, AppRepository, or AppStorage. That's not laziness — it's structural: the engines take AppStorage (a concrete class bound to a Context) and NetworkClient (a concrete OkHttp wrapper) as constructor args, so they can't be instantiated in a pure JVM test without Android.

LayerTest filesVerdict
*Planning objects~40excellent
source/ (NetworkClient, providers)5 + fixturesgood
cleanup/4good
Engines (download/sync/epub/tts)0none
AppRepository / AppStorage0none

The fix — introduce narrow interfaces (ports)

The engines don't need the full AppStorage; they need a handful of operations. Define those as interfaces and have the engines depend on the interface. Then a fake in-memory implementation makes the engine testable in a pure JVM test with MockWebServer.

● Now — concrete deps
class DownloadEngine(
    repository: AppRepository,        // concrete
    network: NetworkClient,           // concrete
)
✦ Proposed — depend on ports
interface DownloadStore {
    fun getStory(id: String): Story?
    fun saveChapter(...): String
    fun mutateQueueInPlace(...): List<DownloadJob>
    // …only the ops download actually uses
}

class DownloadEngine(
    private val store: DownloadStore,
    private val network: NetworkClient,
)

AppStorage would implement DownloadStore (and SyncStore, etc.) — zero behavior change in production, but the engine is now testable with a FakeDownloadStore.

Trade-offs

  • Pro: The actual riskiest code (retry loops, archive-on-removal, queue RMW) finally gets regression coverage.
  • Pro: Interfaces document each engine's real dependencies (currently opaque).
  • Con: Several small interfaces to define. Mitigate by extracting them one engine at a time, starting with StorySyncEngine (it's a pure pipeline — easiest).

First step

Extract a SyncStore interface with the 4 methods StorySyncEngine calls on storage, write a FakeSyncStore, and add StorySyncEngineTest using the existing Royal Road HTML fixtures + MockWebServer. This proves the pattern before applying it elsewhere.

P3 Add storage/restore crash-injection tests Impact Effort ▶

The gap

The durability story is the app's strongest claim — atomic writes, fsync, transactional restore, interrupted-download recovery. But none of it is tested directly, because (per P2) AppStorage needs a Context. The invariants that matter most are exactly the ones that are unverified: "a crash mid-write leaves the previous file intact," "a restore failure leaves the old library untouched," "cold start recovers downloading jobs."

The fix

Two complementary moves:

  1. Robolectric tests for AppStorage (P2's interface extraction makes this trivial) that assert the post-conditions above by simulating a crash (e.g. delete the temp file before rename, corrupt the queue mid-write, half-extract a ZIP).
  2. A small property test over DurableJson: for any envelope, a write interrupted at any point still reads back either the old or the new value — never a partial.

Trade-offs

  • Pro: Turns the app's headline reliability claims into executable guarantees.
  • Con: Adds Robolectric as a test dependency (heavier test runs). Acceptable for the storage package only.

First step

Depends on P2 for AppStorage. Start with one test: write a story, truncate the chapter file mid-write, assert readChapter returns the pre-write content (or null gracefully), never a partial string.

Architecture

P4 Finish the LibraryPresenter migration — it's currently dead code dead code Impact Effort ▶

The gap

LibraryPresenter exists and is fully written — but a search of main/ shows it is never instantiated. Its own KDoc says it's "the intended target for migration" off the inline filtering/sorting that LibraryScreen.showLibrary() still does during render. So the refactor was started, the tested LibraryQuery already backs it, and then it was left disconnected.

The fix

Wire LibraryPresenter into AppContainer (or construct it in showLibrary), and replace LibraryScreen's inline applyFilters() / renderTabGrid() logic with presenter.state(...), binding the returned LibraryUiState to the view tree. The screen keeps its programmatic Views; only where the derived list comes from changes.

Trade-offs

  • Pro: Disk reads / queries move fully off the render path (the documented "Speed S3" goal), and the dead code becomes live.
  • Pro: Makes a future "reactive rebuild only when state changes" (P9) straightforward.
  • Con: Need to preserve the in-place progress patching (patchLibraryProgress) so live download ticks don't snap scroll. Low risk — the presenter derives the list; patching stays a view-tree concern.

First step

This is the single cheapest win in the doc. Construct the presenter in showLibrary, call .state(...) to get the filtered list, and delete the inline query call. Verify the existing LibraryQueryTest still passes (it backs the presenter unchanged).

P5 Unify the network layer behind one typed interface Impact Effort ▶

The gap

NetworkClient is already a good single chokepoint, but its surface is stringly-typed: fetch(url) returns a raw String, retry/rate-limit policy is hardcoded inside it (ScribbleHub-only 403/429 retry, 1500ms gap), and the providers each hand-parse the result with Jsoup inline. There's no shared notion of a typed response, no central place to observe/cancel requests, and the retry policy can't be tuned per-call beyond a timeout.

The fix — a thin response type + policy object

class Response(val status: Int, val body: String, val headers: Headers) {
    fun ensureSuccess(): Response = if (status in 200..299) this else throw HttpException(this)
}

data class RequestPolicy(
    val retryOn: (Int) -> Boolean = { it == 403 || it == 429 },
    val maxAttempts: Int = 3,
    val rateLimitGapMs: Long? = null,
)

This keeps NetworkClient as the implementation but exposes a testable, observable boundary. Providers receive a Response and decide how to parse it; policy moves out of hidden conditionals into data.

Trade-offs

  • Pro: Per-call retry/rate policy becomes explicit and testable; enables P8 (cache) and P12 (observability) cleanly.
  • Pro: No Retrofit needed — the app deliberately avoided it; this stays hand-rolled.
  • Con: Touches every provider's fetch call. Do it incrementally; keep a fetch(url): String compat shim during migration.

First step

Introduce Response + RequestPolicy and have one provider (RoyalRoadProvider) use them behind the scenes, with fetch() delegating. No behavior change; proves the shape.

P6 Move toward immutable models & kotlinx.serialization Impact Effort ▶

The gap

Models are data classes with mutable var fields and MutableLists, chosen so Gson can populate them via reflection. This has real costs: it's why AppRepository needs a monotonic downloadStateVersion counter (model equality is unreliable when fields mutate in place), it invites accidental mutation across components, and Gson's reflection is slower and less safe than compile-time serialization. DownloadJobStatus.wire is a bespoke compat shim that kotlinx.serialization would handle declaratively.

The fix

Two-stage, and explicitly opt-in:

  1. Immutable models first (val, List, .copy() for updates). Engines already mostly do copy-on-write in their Planning objects; promote that to the models. The downloadStateVersion counter can eventually be replaced by real StateFlow equality.
  2. kotlinx.serialization as a follow-up — it generates serializers at compile time, needs no var, and handles the legacy wire strings via @SerialName. Keep Gson for reading old files during a migration window.

Trade-offs

  • Pro: Eliminates whole classes of aliasing bugs; makes StateFlow emission reliable; faster and safer serialization.
  • Con: This is the largest proposal. Gson ↔ kotlinx migration must preserve the existing on-disk JSON exactly (the AGENTS.md flags storage formats as compatibility-sensitive). Mitigate by reading both formats during transition and writing only the new one.
  • Con: Touches every model and every mutation site. Not a first batch item.

Don't attempt this before P2. Without engine tests, a serialization migration is unsafe. Do it last, behind the test coverage P2/P3 add.

First step

Just convert Tab (small, low-coupling) to val fields + .copy() as a proof of concept, keeping Gson. Establish the migration reading-dual-format pattern on one type before widening.

P7 Adopt a typed error model instead of exceptions + string classification Impact Effort ▶

The gap

Errors flow as exceptions that are later re-parsed from strings: DownloadErrorClassifier regex-matches "HTTP\\s+(\\d+)" out of exception messages and substrings like "cancel" / "missing story" to decide retry behavior. parseChapterContent throws on failure specifically so it "surfaces as a retryable download-job error" — meaning control flow rides on exception messages. This is fragile (a message wording change silently breaks classification) and untestable as a unit without constructing exception strings.

The fix

sealed interface FetchError {
    data class Http(val code: Int, val url: String) : FetchError
    data class RateLimited(val retryAfterMs: Long?) : FetchError
    data class Network(val cause: Throwable) : FetchError
    data object ContentTooShort : FetchError
    data object NoProvider : FetchError
}

The network layer returns Result<T> or throws typed errors; DownloadErrorClassifier becomes a pure when over the sealed type — no regex, no string sniffing.

Trade-offs

  • Pro: Classification becomes exhaustive and compile-checked; retry decisions are testable without string games.
  • Pro: Pairs naturally with P5's Response type.
  • Con: Kotlin's Result has ergonomic limits in suspend chains; a sealed FetchError thrown or wrapped in a custom Either is cleaner but adds a type to learn.

First step

Define FetchError, have NetworkClient throw FetchError.Http instead of IllegalStateException("HTTP …"), and rewrite DownloadErrorClassifier to match on it. The existing DownloadErrorClassifierTest makes this a safe, verified change.

Performance

P8 Add an HTTP cache to OkHttp quick win Impact Effort ▶

The gap

The shared OkHttpClient has no cache configured. Every sync re-fetches the metadata page and every re-download of a cleared chapter hits the network fresh. For a story with hundreds of chapters, re-syncing after a data clear means hundreds of uncached GETs, all paying the ScribbleHub 1500ms rate gate.

The fix

val cacheDir = File(context.cacheDir, "okhttp_cache")
val cache = Cache(cacheDir, 50L * 1024 * 1024)  // 50 MiB
val client = OkHttpClient.Builder()
    .cache(cache)
    .connectTimeout(30, SECONDS)
    .build()

Pass the Context into NetworkClient's constructor (it's constructed in AppContainer, which already has one). Use CacheControl.FORCE_NETWORK on sync (you want fresh metadata) but allow the cache to satisfy repeat chapter-content fetches.

Trade-offs

  • Pro: Dramatic speedup on re-downloads and partial retries; less bandwidth; friendlier to the source sites.
  • Pro: Trivial change, no behavior risk.
  • Con: Need cache-control awareness: sync should bypass the cache for metadata; chapter fetches can use it. A per-call Cache-Control header (cheap with P5's policy) handles this cleanly; without P5, add a boolean param.

First step

Add the Cache to the client in AppContainer and verify a re-download of an already-fetched chapter is served from cache in the emulator. This alone is worth doing today.

P9 Tighten the Library render path — diff by identity, not full rebuild Impact Effort ▶

The gap

The Library screen already does in-place progress patching — good — but it still re-derives the full filtered/sorted list on every downloadStateVersion tick, and the diffing is hand-rolled. With mutable models (P6) the diff can't rely on equality, which is why the counter exists. The screen also guards on renderedRoot.parent === frame to avoid patching a torn-down tree — a sign the lifecycle wiring is more manual than it needs to be.

The fix

Once P4 (presenter) is wired, have the presenter expose a StateFlow<LibraryUiState> derived from repository.libraryFlow via .map { derive(it) }. The screen collects it and only re-binds the RecyclerView/GridLayout when the visible summaries actually change (DiffUtil handles row-level animation). The manual counter and the parent-guard both go away.

Trade-offs

  • Pro: Smooth, minimal-overhead updates; removes fragile manual lifecycle guards.
  • Pro: Naturally extends to the Details screen's similar patching logic.
  • Con: Depends on P4 and benefits greatly from P6 (immutable models make DiffUtil correct). Sequence it after both.

First step

After P4, convert LibraryPresenter.state() into a reactive flow and replace the screen's collect-and-patch with a DiffUtil-backed adapter update on a single test story.

P10 Consider WorkManager for opportunistic background sync (not the download loop) Impact Effort ▶

Read carefully — this does not propose replacing the foreground services. The download and TTS loops are correctly foreground services (they need a notification, active progress, and user-initiated lifecycle). This is only about automatic background sync of new chapters while the app is closed.

The gap

Right now a story only gets new chapters when the user opens it and taps Sync. There's no "check for updates overnight" path. For an archiver, stale chapter lists are a common pain point.

The fix

Add a periodic WorkManager PeriodicWorkRequest (e.g. once per 12h, network-unmetered constraint) that calls StorySyncEngine.fetchOrSync for each library story with pendingNewChapterIds or a stale lastUpdated. It only refreshes metadata + TOC — it does not download chapter bodies (that stays user-initiated via the foreground service). WorkManager is the right tool here precisely because it survives reboots and respects doze, which a raw service does poorly.

Trade-offs

  • Pro: Library stays current without manual sync; respects battery constraints.
  • Con: Introduces the one dependency the codebase deliberately avoided. Justify it narrowly: it's only for metadata refresh, gated behind a setting, and the existing engines are reused unchanged.
  • Con: Adds a permission consideration on Android 14+ (FOREGROUND_SERVICE_SPECIAL_USE not needed; WorkManager needs no extra permission for bounded background work).

First step

Add a BackgroundSyncWorker that syncs one story, schedule it on-demand, and verify it completes on the emulator with the screen off. Gate the periodic schedule behind a Settings toggle before enabling by default.

UX & Product

P11 Lower the cost of adding a new novel source Impact Effort ▶

The gap

The SourceProvider abstraction is clean, but adding a site still means writing one fairly large object (RoyalRoadProvider and ScribbleHubProvider are each 90–140 lines) and registering it in the SourceRegistry list. There's no shared base for the common "parse metadata with OG-tag fallbacks" pattern — each provider re-implements the fallback ladder. The user also can't point the app at a new site without a code release.

The fix (two layers)

  1. Developer-facing: Extract a BaseHtmlSourceProvider or a set of reusable helpers (metadataFromOGTags, chaptersFromAnchors(selector), blockText is already shared) so a new site is ~30 lines of selectors + id derivation. Keep the SourceRegistry list but make it the only registration point.
  2. User-facing (optional, larger): A "custom source" feature where the user supplies a URL pattern + CSS selectors (stored as data, not code). This is the legacy app's territory; only pursue if there's real demand, since selector-robustness against site changes becomes a support burden.

Trade-offs

  • Pro (dev-facing): Third/fourth sources become cheap; the OG-fallback logic is tested once.
  • Con: Over-abstracting source parsing can hurt when a site is genuinely different (ScribbleHub's AJAX pagination is already a special case). Keep helpers optional, not mandatory.

First step

Pull the shared metadata-fallback helpers out of the two providers into a SourceHtml object and have both call them. Pure refactor, fully covered by SourceProviderTest.

P12 Add lightweight observability — know what fails in the field quick win Impact Effort ▶

The gap

There is currently no structured logging or telemetry. When a download fails in the field, the error is stored on the DownloadJob (error, errorCategory, errorCode) — which is great for the UI — but there's no aggregate view of which sources/categories dominate, no perf signal on sync/EPUB duration, and no way to debug "why did TTS stop" beyond the persisted session. Developent relies on reading logcat on the emulator.

The fix

A single, tiny, dependency-free Metrics object that records structured events (counters + timings) into a ring buffer in memory and optionally a JSON log file. No network, no analytics SDK. Key signals:

  • Per-source fetch success/failure counts by FetchError category (ties into P7)
  • Sync and EPUB generation durations (histogram buckets)
  • TTS playback interruptions and resume outcomes
  • Storage write latencies (to catch the P1 monitor contention if it appears)

Expose it on a hidden Settings screen ("Diagnostics") for now. If external reporting is ever wanted, the structured events are ready to ship to any sink.

Trade-offs

  • Pro: Turns "works on the emulator" into "we know what fails in production" without any privacy cost (local-only by default).
  • Pro: Cheap, and it directly informs whether P1/P5/P7 actually moved the needle.
  • Con: Discipline required to keep it from becoming noise. Keep the event vocabulary small and typed.

First step

Add a Metrics object with counter(name, tags) + timing(name, ms), wire it into DownloadEngine.handleJobError and StorySyncEngine.fetchOrSync, and render the buffer on a new Settings row. Half a day, immediately useful.

14Suggested Sequence

A dependency-aware ordering. Each wave is independently shippable; later waves build on earlier ones but don't strictly require them.

Wave 1 Quick wins & confidence

Lowest risk, immediately useful, and unblocks everything else by proving the codebase is observable and tested.

  • P8 — HTTP cache
  • P4 — Finish LibraryPresenter
  • P12 — Observability

Wave 2 Correctness & testability

Fixes the latent reliability gaps and builds the test coverage that makes Wave 3 safe.

  • P1 — suspend + Mutex
  • P2 — Engine test ports
  • P3 — Crash-injection tests
  • P7 — Typed error model

Wave 3 Structural evolution

Larger, only attempt once Wave 2's tests are green. Each is optional and independent.

  • P5 — Typed network layer
  • P9 — Reactive render path
  • P6 — Immutable models + serialization
  • P11 — Source abstraction
  • P10 — Background sync (optional)

None of these require changing the app's defining choices. The single-activity, no-XML, no-DI-framework, foreground-service architecture stays intact throughout. Every proposal is an evolution within the existing model.

15Decision Principles

When evaluating any of these (or future proposals), the codebase's own documented values are the tiebreaker.

Keep

  • The planning/engine split — push logic into pure, tested *Planning objects.
  • Single-owner storage + monitor serialization — the model is right; only the await mechanism is in question (P1).
  • No XML UI — lean into the programmatic DSL.
  • Durability-first writes — atomic + fsync, always.
  • Foreground services for user-initiated work.

Change only with justification

  • Adding a DI framework — the manual AppContainer is fine at this scale.
  • Jetpack Compose — the View DSL works; a migration would be enormous for unclear gain.
  • Room/SQLite — only if JSON storage genuinely hits a scaling wall (it hasn't).
  • Retrofit — P5 keeps the network layer hand-rolled and typed.
  • WorkManager — only for opportunistic background sync (P10), never the loops.

The guiding question for any change: "Does this move deterministic logic into a pure, tested *Planning object, or move I/O out of the render path?" If yes, it fits the architecture. If it adds a framework to avoid writing a class, it probably doesn't.
